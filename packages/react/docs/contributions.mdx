# Contributing to F0

F0 is Factorial's design system. Anyone at Factorial can propose, build, and ship components — but every component follows the same lifecycle so the system stays coherent, reliable, and easy to consume.

This page is the entry point. It tells you **where to start** depending on what you want to do, and links to the documents that govern each step.

Reach us in [#f0 Slack channel](https://factorialteam.slack.com/archives/C06QT46115K) — we're happy to support you.

---

## I want to…

### …propose a new component or pattern

> **Every proposal is evaluated on its merits.** F0 grows intentionally — we say yes when a component has clear cross-team value, and we say no with a documented reason and an alternative (extend an existing component, app-local solution, etc). Use Claude/OpenCode (or Storybook search) to walk through [Phase 0 — Discovery & Evaluation](https://github.com/factorialco/factorial-one/blob/main/packages/react/.skills/f0-component-contribution/SKILL.md#phase-0--discovery--evaluation-before-proposing) **before** opening the proposal: does it already exist? can something be extended? do at least 2 different teams need it? could a pattern solve it?

1. Open a [**Component proposal** issue](https://github.com/factorialco/factorial-one/issues/new?template=component-proposal.yml). The template walks you through Phase 0 in-line.
2. An F0 designer reviews the proposal and either approves it (checks the **"Reviewed and approved"** box on the issue), requests more evidence, or declines with a documented reason. Proposals need **≥2 real use cases from different teams**, but a designer can waive the team-count requirement for clearly foundational primitives.
3. Once approved, follow the [`f0-component-contribution`](https://github.com/factorialco/factorial-one/blob/main/packages/react/.skills/f0-component-contribution/SKILL.md) skill to design and build it in `experimental/`. See the [Where it goes](/?path=/docs/lifecycle-where-it-goes--docs) page to know which destination folder applies once promoted.

### …improve or extend an existing component

1. Open a [**Component enhancement** issue](https://github.com/factorialco/factorial-one/issues/new?template=component-enhancement.yml) describing the change and its impact.
2. If the component is **stable**, the change goes through the same design review as a new component (see [Design Review](/?path=/docs/lifecycle-design-review--docs)).
3. If the component is **experimental**, the owning team can iterate freely.

### …report a bug

Open a [**Bug report** issue](https://github.com/factorialco/factorial-one/issues/new?template=bug-report.yml). Bugs are routed to the component's owner (see [CODEOWNERS](https://github.com/factorialco/factorial-one/blob/main/CODEOWNERS)).

### …promote an experimental component to stable

Follow the [`f0-component-promotion`](https://github.com/factorialco/factorial-one/blob/main/packages/react/.skills/f0-component-promotion/SKILL.md) skill. Promotion is owned by Foundations and gated by the [Definition of Done](/?path=/docs/lifecycle-definition-of-done--docs).

### …deprecate or remove a component

See the [Deprecation & Removal policy](/?path=/docs/lifecycle-release-and-versioning--docs).

## The lifecycle in one diagram

```
1. Problem        →  2. Design        →  3. Build           →  4. Real use       →  5. Promote
   (issue +          (design review,     (experimental/,        (≥3 teams in         (Foundations
    designer          Figma, API)         owned by team)         production)          moves to stable)
    review)
```

Each phase has clear entry and exit criteria. The full contract lives in the [Definition of Done](/?path=/docs/lifecycle-definition-of-done--docs).

## Maturity levels

F0 has **three** maturity levels: **Experimental**, **Stable**, and **Deprecated**. See the [Component Maturity model](/?path=/docs/lifecycle-components-maturity--docs) for what each one guarantees and what it requires.

> `internal` is **not** a maturity level — it's a Storybook tag used to hide stories from the public sidebar. The components in `src/ui/` (Radix wrappers used only inside F0) live under the **UI Wrappers** sidebar group. See the [Where it goes](/?path=/docs/lifecycle-where-it-goes--docs) page for placement rules.

---

## Ownership

- **Foundations team** owns: `components/`, `patterns/`, `ui/`, `kits/`, `layouts/`, `hooks/`.
- **Satellite Design System (SDS) teams** own: `sds/<area>/`.
- **Experimental components** are owned by the **team that created them** until promotion.
- Promotion to stable transfers ownership to **Foundations**.

See [`CODEOWNERS`](https://github.com/factorialco/factorial-one/blob/main/CODEOWNERS) for the live mapping.
