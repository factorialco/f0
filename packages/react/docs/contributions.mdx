# Contributing to F0

F0 is Factorial's design system. Anyone at Factorial — designer, product engineer, or Foundations — can shape it. But F0 grows **intentionally**: we add things when they have clear cross-team value, and we say no with a documented reason and an alternative.

**The door is a conversation.** You start by talking to an AI agent (Claude / OpenCode) and, when needed, to us in [#f0-support](https://factorialteam.slack.com/archives/C082ZNKS403). **The amount of process scales with the impact of what you're doing.**

> You don't need to know where your idea fits or what "Core vs Kit vs Domain specific" means before you start. The agent figures that out with you. This page just maps what happens so it isn't a black box.

---

## Start here: talk to Claude

Open Claude (or OpenCode) in the f0 repo — it has the F0 skills loaded — and describe what you need in plain terms: _"I need a component that lets a user pick a date range."_

> New to the repo? Get **Claude Code** and ask it to set up the f0 repo for you — it runs the setup itself, so there's nothing manual to keep track of.

The agent will **search the repo and Storybook for you** and tell you which of these you're in:

| The agent finds…                                                 | What you do                              | Artifact |
| ---------------------------------------------------------------- | ---------------------------------------- | -------- |
| A component (or a composition of existing ones) covers your case | **Use it.** Done.                        | none     |
| Nothing covers it, and it's one screen with no reuse or identity | **It lives in your product**, not F0.    | none     |
| Nothing covers it, and it's a real, reusable need                | **It's a contribution** → keep reading.  | PR       |

When a row above resolves it, you're done right there. That's the point of this first step: it removes work that shouldn't happen before anyone invests in it.

## The flow, once it's a real contribution

Three steps. Only the last one always produces an artifact (the PR).

### 1 · Review & filter — _is it already there, and should it be in F0?_

The agent **reviews what already exists** — searching the repo and Storybook — then filters. If it already exists, is covered by extending something, or belongs in your product, you're already done. Only genuinely new, reusable needs continue.

### 2 · Align — _what is it, and where does it live?_

You raise it as a message in [#f0-support](https://factorialteam.slack.com/archives/C082ZNKS403) with whatever you have — the need, a Figma, a [Composer](/?path=/docs/lifecycle-getting-started--docs) prototype, or a draft PR. Foundations confirms the **systemic fit** and the home.

First: is it **new**, or an **extension** of something that exists? That decides where it goes:

- An **extension** (a prop, a variant on an existing component) lives **where that component already is** — you extend it, you don't create a new home.
- Something **new** picks its home: **Core**, a **Kit**, or **Domain specific** (see [Where it goes](/?path=/docs/lifecycle-where-it-goes--docs)).

How much extra validation it needs is covered below. Foundations responds as availability allows — there's **no fixed SLA**. If you're blocked, say so in the thread and it gets prioritized.

### 3 · Build to the standard — _the PR_

You (with Claude and the F0 skills) build it in `experimental/` to the [Definition of Done](/?path=/docs/lifecycle-definition-of-done--docs): F0-prefixed named exports, design tokens, i18n, a11y, Storybook story, unit tests, MDX docs. The [`f0-component-contribution`](https://github.com/factorialco/factorial-one/blob/main/packages/react/.skills/f0-component-contribution/SKILL.md) skill drives this end to end.

**Every new or changed thing lands as a PR** — that's where code review happens and where it merges. It ships as **experimental** first. Later, once it's adopted and its API has settled, **Foundations promotes it to stable** (see [`f0-component-promotion`](https://github.com/factorialco/factorial-one/blob/main/packages/react/.skills/f0-component-promotion/SKILL.md)).

## How much validation a change needs

**Everything that enters F0 is reviewed — always.** Whatever the maturity, two things happen: Foundations confirms in [#f0-support](https://factorialteam.slack.com/archives/C082ZNKS403) that it belongs and where (the alignment above), and the PR goes through **code review + `f0-quality-gate`**. A **new component** also gets a **design review**. _("Iterating freely" only means the owner can change an already-**experimental** component's API without a fresh design review — experimental is unstable by design. It still entered through the same review.)_

Beyond that, how much _extra_ validation a change needs comes down to **two questions**:

1. **Is it already in use?** Changing something other people already consume is high-impact — it needs a **migration plan** (deprecation window, migration guide, consumer notice, version bump — see [Release & Versioning](/?path=/docs/lifecycle-release-and-versioning--docs)). Adding something new rarely ripples.
2. **Is it shared vocabulary, or a new public API?** A new or changed **token** or **icon**, or a new component's API, gets a **design review** — consistency there is expensive even when nothing breaks. A backward-compatible prop on an experimental component doesn't.

> **Adding is never "free".** Something that duplicates or re-solves what already exists fragments the system — a real cost, just not a _breaking_ one. That's what the [Review & filter](#the-flow-once-its-a-real-contribution) step and "extend before you create" are for.

## What if it's not a straight yes?

The flow isn't only "enters F0 or disappears." Alignment can end in one of these — and **each has a next step**, none is a dead end:

- **Iterate** — Foundations asks for changes or more evidence. → **Refine and bring it back to the same thread.** You pick up where you left off; nothing is thrown away.
- **Redirected** — it's real, but it belongs in a **Kit**, as **Domain specific**, or as an **extension** of something existing rather than a new component. → **Re-enter the flow at that home** — the agent points you to the exact place, and it's often a smaller change than you thought.
- **Lives in your product** — bespoke UI for one screen, with no reuse and no shared identity, belongs in your product, not F0. → **Build it there with Claude / Composer, assembled from F0 pieces** — you're not on your own; it just doesn't live in F0. A **valid outcome**, not a rejection — F0 stays lean because of it.
- **Declined** — it doesn't fit F0's strategy. → You get a **documented reason and, where possible, a concrete alternative.** That's the end of the F0 path.

## Reporting a bug

Bugs don't go through the contribution flow. Report them in [#f0-support](https://factorialteam.slack.com/archives/C082ZNKS403) — they're routed to the component's owner (see [CODEOWNERS](https://github.com/factorialco/factorial-one/blob/main/CODEOWNERS)). Or, if you can, fix it and open a PR against the owner directly.

## Maturity levels

F0 has **three** maturity levels: **Experimental**, **Stable**, and **Deprecated**. See the [Component Maturity model](/?path=/docs/lifecycle-components-maturity--docs) for what each one guarantees and what it requires.

> `internal` is **not** a maturity level — it's a Storybook tag used to hide stories from the public sidebar. The components in `src/ui/` (Radix wrappers used only inside F0) live under the **UI Wrappers** sidebar group. See the [Where it goes](/?path=/docs/lifecycle-where-it-goes--docs) page for placement rules.

## Who does what

F0 is built and maintained by the **Foundations team**, with contributions from every product team. Three roles matter in the flow:

- **Foundations** — owns Core (`components/`, `patterns/`, `ui/`, `kits/`, `layouts/`, `hooks/`) and the shared **tokens** and **icons**. They're the only ones who **approve** what enters F0, **mark a component stable**, and **promote** it. Reach them in [#f0-support](https://factorialteam.slack.com/archives/C082ZNKS403).
- **Your team** — builds the contribution and **owns and maintains it**. While it's experimental, your team is its **CODEOWNER** (add its path to [`CODEOWNERS`](https://github.com/factorialco/factorial-one/blob/main/CODEOWNERS)) and fields its bugs. On promotion, a Core/Kit component's ownership transfers to Foundations; a **Domain specific** component stays yours.
- **Claude / OpenCode** — the **access layer**: it searches, filters, guides placement, and builds to the standard. It does **not** approve, mark anything stable, or merge on your behalf — those are human / Foundations calls.

**Source of truth:** the repo — code, [`CODEOWNERS`](https://github.com/factorialco/factorial-one/blob/main/CODEOWNERS), and these docs — is authoritative. If Claude ever seems to contradict them, the repo wins; treat Claude's output as a suggestion to verify, not a ruling.
