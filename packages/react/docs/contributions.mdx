# Contributing to F0

F0 is Factorial's design system. Anyone at Factorial — designer, product engineer, or Foundations — can shape it. But F0 grows **intentionally**: we add things when they have clear cross-team value, and we say no with a documented reason and an alternative.

**The door is a conversation, not a form.** You start by talking to an AI agent (Claude / OpenCode) and, when needed, to us in [#f0-support](https://factorialteam.slack.com/archives/C082ZNKS403). **The amount of process scales with the impact of what you're doing** — most requests are resolved in that conversation, with nothing to file.

> You don't need to know where your idea fits or what "Core vs Kit vs Domain specific" means before you start. The agent figures that out with you. This page just maps what happens so it isn't a black box.

---

## Start here: talk to Claude

Open Claude (or OpenCode) in the f0 repo — it has the F0 skills loaded — and describe what you need in plain terms: _"I need a component that lets a user pick a date range."_

The agent will **search the repo and Storybook for you** and tell you which of these you're in:

| The agent finds…                                              | What you do                                        | Artifact |
| ------------------------------------------------------------- | -------------------------------------------------- | -------- |
| It already exists                                             | **Use it.** Done.                                  | none     |
| An existing component covers it by adding a prop / variant    | **Extend it** (see _Evolving something_ below).    | PR       |
| It's specific to one screen, no reuse, no shared identity     | **It lives in your product**, not F0.              | none     |
| It's a real, reusable need not yet solved                     | **It's a contribution** → keep reading.            | PR       |

**Most requests end in the first or third row** — resolved in the conversation, nothing to file. That's the point: the filter removes work that shouldn't happen before anyone invests in it.

---

## The flow, once it's a real contribution

Three steps. Only the last one always produces an artifact (the PR).

### 1 · Filter — _should this be built in F0?_

Answered above, with the agent. If it exists, extends, or belongs in your product, you're already done. Only genuinely new, reusable needs continue.

### 2 · Align — _what is it, and where does it live?_

Bring **whatever you have** to [#f0-support](https://factorialteam.slack.com/archives/C082ZNKS403): just the need, a Figma, a [Composer](/?path=/docs/lifecycle-getting-started--docs) prototype, or a draft PR. Foundations confirms the **systemic fit** — does it belong, and where does it live: **Core**, a **Kit**, or **Domain specific** (see [Where it goes](/?path=/docs/lifecycle-where-it-goes--docs)).

This is a validation, **not paperwork**. There is no GitHub issue to file and no checkbox to wait on. How deep the alignment goes **scales with impact**:

- **Low impact** (a prop, a variant) → agree in the thread and go.
- **Medium** (a new component) → a quick design + placement check.
- **High impact** (a new or changed **token**, a new **icon**, or a **pattern** other teams will adopt) → an explicit design validation up front, _before_ you build, because it touches the base and other teams. This is where we slow down on purpose.

> For now, all of this happens in [#f0-support](https://factorialteam.slack.com/archives/C082ZNKS403) — there's no separate form or GitHub issue for you to file. If a proposal needs tracking, it's raised from the thread.

### 3 · Build to the standard — _the PR_

You (with Claude and the F0 skills) build it in `experimental/` to the [Definition of Done](/?path=/docs/lifecycle-definition-of-done--docs): F0-prefixed named exports, design tokens, i18n, a11y, Storybook story, unit tests, MDX docs. The [`f0-component-contribution`](https://github.com/factorialco/factorial-one/blob/main/packages/react/.skills/f0-component-contribution/SKILL.md) skill drives this end to end.

**Every new or changed thing lands as a PR** — that's where code review happens and where it merges. It ships as **experimental** first. Later, once it's adopted and its API has settled, **Foundations promotes it to stable** (see [`f0-component-promotion`](https://github.com/factorialco/factorial-one/blob/main/packages/react/.skills/f0-component-promotion/SKILL.md)).

## Evolving something that already exists

Most contributions aren't net-new components — they're **changes to what's already there**. Same door (talk to Claude first), but the path and the amount of validation depend on **what** you're changing and **whether it's in use**:

| Change                                         | Impact (blast radius)        | Design check?                          | What it takes |
| ---------------------------------------------- | ---------------------------- | -------------------------------------- | ------------- |
| Add a **prop / variant** (backward-compat)     | Low                          | Only if the component is **stable**    | Tests + story + docs updated. Experimental → the owner iterates freely. |
| Add a new **component**                        | Low                          | Yes — it's a new public API            | The full build (see the flow above). |
| Add a new **icon**                             | Low                          | Yes — consistency with the icon set    | Additive; follow the repo's icon-generation flow. |
| Add a new **token**                            | Low–medium                   | Yes — avoid overlapping an existing one | Additive, but tokens are shared vocabulary. |
| **Change** a token or icon **already in use**  | **High** → migration         | Yes                                    | Ripples through every consumer; needs a migration plan. |
| **Iterate** a **pattern** in use               | **High** → consumers migrate | Yes                                    | Align widely before building. |
| Any **breaking change** to something in use    | **High** → migration         | Yes                                    | Deprecation window, migration guide, consumer notice, version bump — see [Release & Versioning](/?path=/docs/lifecycle-release-and-versioning--docs). |

The driver is one question: **are you changing something others already use?** Adding something new rarely ripples — it's the *changes to what's in use* that carry high impact and may force a migration. Shared visual vocabulary (tokens, icons) and new public APIs also get a **design check**, because inconsistency there is expensive even when nothing breaks.

## What if it's not a straight yes?

The flow isn't only "enters F0 or disappears." Alignment can end in one of these — and **each has a next step**, none is a dead end:

- **Iterate** — Foundations asks for changes or more evidence. → **Refine and bring it back to the same thread.** You pick up where you left off; nothing is thrown away.
- **Redirected** — it's real, but it belongs in a **Kit**, as **Domain specific**, or as an **extension** of something existing rather than a new component. → **Re-enter the flow at that home** — the agent points you to the exact place, and it's often a smaller change than you thought.
- **Lives in your product** — bespoke UI for one screen, with no reuse and no shared identity, belongs in your product, not F0. → **Build it there with Claude / Composer, assembled from F0 pieces** — you're not on your own; it just doesn't live in F0. A **valid outcome**, not a rejection — F0 stays lean because of it.
- **Declined** — it doesn't fit F0's strategy. → You get a **documented reason and, where possible, a concrete alternative.** That's the end of the F0 path.

## Reporting a bug

Bugs don't go through the contribution flow. Report them in [#f0-support](https://factorialteam.slack.com/archives/C082ZNKS403) — they're routed to the component's owner (see [CODEOWNERS](https://github.com/factorialco/factorial-one/blob/main/CODEOWNERS)). Or, if you can, fix it and open a PR against the owner directly.

```
     ┌─ use it / extend / lives in your product ─┐   (most requests end here, no artifact)
     │                                            │
1. Filter ──▶ 2. Align ──────▶ 3. Build ──▶ Review ──▶ Merge ──▶ Experimental ──▶ Promote
  (with        (#f0-support,   (experimental/,  (PR)              (in real use)    (Foundations

```

Each phase has clear entry and exit criteria. The full contract lives in the [Definition of Done](/?path=/docs/lifecycle-definition-of-done--docs).

## Maturity levels

F0 has **three** maturity levels: **Experimental**, **Stable**, and **Deprecated**. See the [Component Maturity model](/?path=/docs/lifecycle-components-maturity--docs) for what each one guarantees and what it requires.

> `internal` is **not** a maturity level — it's a Storybook tag used to hide stories from the public sidebar. The components in `src/ui/` (Radix wrappers used only inside F0) live under the **UI Wrappers** sidebar group. See the [Where it goes](/?path=/docs/lifecycle-where-it-goes--docs) page for placement rules.

---

## Ownership

- **Foundations team** owns: `components/`, `patterns/`, `ui/`, `kits/`, `layouts/`, `hooks/` — and the shared **tokens** and **icons**.
- **Domain teams** own their **Domain specific** components: `sds/<area>/`.
- **Experimental components** are owned by the **team that created them** until promotion.
- Promotion to stable transfers ownership to **Foundations**.

See [`CODEOWNERS`](https://github.com/factorialco/factorial-one/blob/main/CODEOWNERS) for the live mapping.
