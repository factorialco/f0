# Definition of Done

This is the **technical contract** for an F0 component. Every component, regardless of who builds it, must satisfy these criteria to enter and stay at each maturity level. It is the source of truth referenced by the contribution and promotion skills, the design review, and the quality gate.

For the human entry-point and "where do I start", see [Contributions](/?path=/docs/lifecycle-how-to-contribute--docs).

---

## Lifecycle phases

| Phase            | Owner                  | Output                                                  |
| ---------------- | ---------------------- | ------------------------------------------------------- |
| 1. Problem       | Proposer               | Alignment in #f0-support + Foundations decision         |
| 2. Design        | Proposer + Foundations | Figma design + API draft + design review                |
| 3. Build         | Proposer's team        | Component in `experimental/` with stories, tests, docs  |
| 4. Real use      | Proposer's team        | Adoption by ≥3 product teams in production              |
| 5. Promote       | Foundations            | Component moved to `components/` (or other stable zone) |

---

## Phase 1 — Problem

**Entry:** anyone identifies a recurring UI need and, with the F0 agent, has filtered it (it doesn't already exist, isn't covered by extending something, and belongs in F0). See [`f0-component-contribution`](https://github.com/factorialco/factorial-one/blob/main/packages/react/.skills/f0-component-contribution/SKILL.md).

**Exit (DoD):**

- Alignment reached with Foundations in `#f0-support` — **that's where a proposal is raised, for now; no GitHub issue template to file yourself.** Foundations confirms the systemic fit and the destination, with one of:
  - `proceed` — fits F0; destination confirmed (**Core**, a **Kit**, or **Domain specific** `sds/<area>/`). → go to Phase 2 (Design), then Build.
  - `redirect` — real, but belongs elsewhere (Kit / Domain specific / an extension of something existing). → re-enter the flow at that placement.
  - `needs-info` — not enough to decide. → clarify in the thread, then re-align (loop).
  - `lives-in-product` — no cross-domain reuse or shared identity; it stays in the product monolith. → build it there (Claude / Composer can still assemble it from F0 pieces). A valid outcome, not a rejection.
  - `declined` — doesn't fit F0 strategy. → documented reason + a concrete alternative where possible. End of the F0 path.

---

## Phase 2 — Design

**Entry:** alignment reached — the need is accepted for Core, a Kit, or Domain specific.

**Exit (DoD):**

- Figma design exists and is linked from the issue.
- A draft API (props, variants, slots) is documented in the issue.
- A **design review** has been completed (see [Design Review](/?path=/docs/lifecycle-design-review--docs)) and all blocking comments are resolved.

---

## Phase 3 — Build

**Entry:** design review approved.

**Exit (DoD):**

- Component lives in `src/experimental/<Category>/F0Name/` (or `src/sds/<area>/` for domain). See the [Where it goes](/?path=/docs/lifecycle-where-it-goes--docs) page to confirm the right subfolder by intended destination.
- Story file with `tags: ["autodocs", "experimental"]`.
- Unit tests (Vitest) cover the public API.
- Storybook play function covers the primary user flow.
- Axe a11y test passes.
- MDX documentation reaches at least the **Acceptable** tier (see [`f0-docs`](https://github.com/factorialco/factorial-one/blob/main/packages/react/.skills/f0-docs/SKILL.md)).
- `f0-quality-gate` skill passes (typecheck, lint, format, tests).
- PR merged.

---

## Phase 4 — Real use

**Entry:** component released as `experimental` for at least one F0 version.

**Exit (DoD) — two axes + re-confirmation:**

**Demand axis (must hit all):**

- Used by **≥3 product teams** in production (verified via cross-repo search in `factorialco` GitHub org).
- No "would you remove this?" pushback from consumers in the last 30 days.

**Maturity axis (must hit all):**

- **No breaking changes for ≥60 days.**
- **Zero open critical or high-severity bugs.**
- `f0-quality-gate` passes including extended subagent reviews (a11y, design tokens, dark mode).
- Component owner declares the API stable in the #f0-support promotion thread.

**Foundations re-confirmation:**

- Foundations team explicitly approves promotion (no auto-promotion).

> There is **no time minimum** beyond the 60-day no-breaking-changes window. A widely-adopted component can promote in 2 months; a niche one may stay experimental for a year.

---

## Phase 5 — Promote

**Entry:** Phase 4 DoD met.

**Exit (DoD):**

- Component moved to `src/components/` (or `patterns/`, `kits/`, `layouts/`, `hooks/`) following the [`f0-experimental-component-migration`](https://github.com/factorialco/factorial-one/blob/main/packages/react/.skills/f0-experimental-component-migration/SKILL.md) skill.
- Story `tags` updated to `["autodocs", "stable"]`.
- MDX docs at **Good** tier minimum (Gold encouraged).
- Ownership transferred to Foundations in `CODEOWNERS`.
- Promotion announced in `#f0-support` Slack and changelog.

**Response time:** an F0 designer + Foundations engineer respond to promotion requests based on team availability and as quickly as possible — we don't commit to a fixed SLA today, as it depends on team size and current focus. For urgent or blocking cases, ping `#f0-support` in Slack.

---

## Maturity levels

Three levels apply to every F0 component:

| Level            | API stability                 | Owner        | Folder                                 |
| ---------------- | ----------------------------- | ------------ | -------------------------------------- |
| **Experimental** | May change                    | Creator team | `src/experimental/`, `src/sds/<area>/` |
| **Stable**       | Backwards-compat only         | Foundations  | `src/components/`, `patterns/`, etc.   |
| **Deprecated**   | Frozen, scheduled for removal | Owner        | Anywhere; flagged in JSDoc             |

> `internal` (the Storybook tag and the `src/ui/` folder) is **not** a maturity level. See [Component Maturity](/?path=/docs/lifecycle-components-maturity--docs).

---

## Deprecation

To deprecate, add three JSDoc tags above the export:

```ts
/**
 * @deprecated Use F0NewComponent instead.
 * @removeIn 2.0.0
 * @migration https://github.com/factorialco/factorial-one/blob/main/packages/react/docs/migrations/f0-old-to-new.md
 */
```

Removal: minimum 1 quarter (≥90 days) after deprecation. Full policy: [Deprecation & Removal](/?path=/docs/lifecycle-release-and-versioning--docs).
