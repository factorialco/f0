Overview of all F0 components, their API stability, and documentation coverage.

## How to read this table

| Column         | Description                                                                                                                                            |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Component**  | Component name as it appears in Storybook                                                                                                              |
| **Zone**       | Source folder: `components` (stable public), `patterns` (composite), `sds` (Domain specific), `kits`, `layouts`, `experimental`, `deprecated` |
| **API Status** | Derived from the `tags` array in the story file. If no status tag is set, it shows as "No tag"                                                         |
| **Unit Tests** | Whether a `__tests__/` folder or `.test.tsx` file exists near the component                                                                            |
| **MDX Docs**   | Whether an `.mdx` documentation file exists alongside the stories                                                                                      |

> The **Internal** badge in the API Status column refers to components in `src/ui/` (Radix wrappers used only inside F0). It is **not** a maturity level â€” it's a folder convention. The three maturity levels are **Experimental**, **Stable**, and **Deprecated** (see [Component Maturity](/?path=/docs/lifecycle-components-maturity--docs)).

## How to update a component's status

Edit the `tags` array in the component's `.stories.tsx` file:

```tsx
const meta: Meta<typeof F0MyComponent> = {
  tags: ["autodocs", "stable"], // or "experimental", "deprecated"
}
```

The data is computed automatically from the `src/` tree on every build and
dev-server start (via the `virtual:f0-component-status-data` Vite/esbuild plugin
in `scripts/component-status-build.mjs`) â€” there is no committed snapshot to
regenerate. Restart the dev server to pick up newly added stories, tests, or
docs.

## Definition of Done (stable)

The **effective status** is what matters: a component counts as **stable** only
when it is both tagged `stable` **and** clears every item below. Anything else â€”
untagged, tagged stable but below the bar, or meeting the bar without the tag â€”
is treated as **experimental**. This mirrors the
[Lifecycle Definition of Done](/?path=/docs/lifecycle-definition-of-done--docs).

| Requirement                | How it's checked                                                     |
| -------------------------- | ------------------------------------------------------------------- |
| Has Storybook stories      | A `.stories.tsx` file exists                                         |
| Has unit tests             | A `__tests__/` folder or `.test.tsx` file exists near the component  |
| Has a play function        | The story defines a `play:` interaction test                        |
| Has MDX documentation      | An `.mdx` docs page exists alongside the stories                     |
| Docs reach `good` quality  | MDX adds DoDonts, a "when not to use" section, and 3+ named examples on top of the Acceptable base (sections + props table) |

> The doc bar is the **Good** tier â€” Promotion to stable requires it (Acceptable
> is only the experimental Build bar). Tier heuristic: `none` â†’ `stub` â†’
> `acceptable` â†’ `good` â†’ `gold` (a coarse structural check, not a prose review).
>
> Not automated here (still manual promotion gates): the axe **a11y** test
> passing, adoption by **â‰Ą3 product teams**, **no breaking changes for 60 days**,
> and Foundations approval.

## Programmatic API

Query any component's status and what it's still missing to be stable, by name.
In the Storybook **preview** console:

```js
// Forgiving name matching: "Button", "F0Button", "button" all work.
f0ComponentStatus.get("Button")
// â†’ { name, zone, apiStatus, docQuality, requirements, missing, meetsBar,
//     taggedStable, stableReady, discrepancy, ... }

f0ComponentStatus.get("F0Alert").missing
// â†’ [] when it meets the bar, else e.g. ["Has MDX documentation", ...]

// Components that claim to be stable but fall below the raised bar:
f0ComponentStatus
  .getAll()
  .filter((c) => c.discrepancy === "tagged-but-below-bar")

// Components that already meet the bar but aren't tagged stable yet:
f0ComponentStatus
  .getAll()
  .filter((c) => c.discrepancy === "meets-bar-not-tagged")
```

The same functions are published as a subpath export, so any consumer can import
them:

```ts
import {
  getComponentStatus,
  getAllComponentStatuses,
  STABLE_REQUIREMENTS,
} from "@factorialco/f0-react/dist/component-status"

const { missing, meetsBar } = getComponentStatus("F0DatePicker")!
```

Inside this repo (stories, tests, tooling) import it from source instead:

```ts
import { getComponentStatus } from "@/component-status"
```

> The dataset is recomputed from source on every build (and dev-server start),
> so the published entry always reflects the source tree at build time â€” there is
> no committed snapshot to keep in sync.

`STABLE_REQUIREMENTS` is the single source of truth for the checklist â€” extend
it there (e.g. to add an a11y gate) and every consumer updates automatically.

### Rendered panel

All display text (badge label, summary line, checklist labels and hints) is
part of the status data, so the panel looks identical in Storybook and in any
consuming app. Drop in the exported `ComponentStability` component with a
component name:

```tsx
import { ComponentStability } from "@factorialco/f0-react/dist/component-status"

;<ComponentStability componentName="F0Card" />
```

It renders nothing when the name doesn't resolve to a tracked component. The
status block shown on every component docs page above is this exact component.
