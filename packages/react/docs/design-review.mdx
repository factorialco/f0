# Design Review

A design review is a synchronous (or asynchronous) checkpoint between the proposer and the Foundations design team, held in **Phase 2** of the lifecycle (see [Definition of Done](/?path=/docs/lifecycle-definition-of-done--docs)).

It is **not** a code review and **not** a quality gate — those happen in Phase 3 via the `f0-quality-gate` skill. The design review focuses on the **subjective and systemic** decisions that automation cannot check.

<Callout tone="hot" title="Design review is NOT where we decide whether it should exist">
That's decided earlier — in the Review & filter step and the alignment with Foundations. By the time something reaches design review, the question is "is this design correct?", not "should we build this?".
</Callout>

---

## When it's required

- Every **new** component (Phase 2).
- Every change to a **stable** component that affects its API, visual design, or behavior.
- Optional for changes to experimental components (the owning team can iterate freely).

---

## Who attends

- **Proposer** — presents the problem, design, and proposed API.
- **At least one Foundations designer** — owns approval.
- **Optional:** an a11y specialist for components with complex interactions.

Async review (comments on the design — a Figma, a Composer prototype, or another — plus the API spec) is acceptable for small changes.

---

## The 10-category checklist

Every reviewer evaluates these categories. Any blocking comment must be resolved before approval.

<Unstyled>
  <Grid min={280} items={[
    { label: "1 · Problem clarity", text: "Is the problem clear? Evidence of recurring need across teams?" },
    { label: "2 · System fit", text: "Coherent with similar F0 components; final placement confirmed (Core / Kit / Domain specific)." },
    { label: "3 · API design", text: "Clear, conventional prop names; discriminated unions for exclusive props; sensible defaults." },
    { label: "4 · Composition", text: "Composes with existing layouts, dialogs, forms; slots where flexibility is needed." },
    { label: "5 · Variants & density", text: "Minimal, meaningful variants; size / density / spacing via tokens." },
    { label: "6 · Visual design", text: "F0 design tokens (no hex); dark mode correct; F0 icons." },
    { label: "7 · Accessibility", text: "Keyboard nav, ARIA, focus order + focusRing(), reduced-motion." },
    { label: "8 · Internationalization", text: "Strings via useI18n(); pluralization; RTL layout." },
    { label: "9 · States & edge cases", text: "Loading, error, empty, disabled; skeleton if data-driven; overflow handled." },
    { label: "10 · Documentation plan", text: "MDX tier declared; ≥3 example stories planned." },
  ]} />
</Unstyled>

---

## Output

A design review ends with one of:

- **Approved** — proceed to Phase 3 (Build).
- **Approved with comments** — proceed; address comments in build.
- **Needs revision** — address blocking comments and re-review.

<Callout>
The whole design conversation and its decision live in the #f0-support thread — not as PR comments. The design can be just a need, a Figma, or a prototype, so there may be no PR yet; the code review on the PR comes later and is separate.
</Callout>
