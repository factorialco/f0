# Decisions

Heuristics for picking between patterns. When in doubt, return to
<a target="_parent" href="/?path=/docs/experimental-crud-patterns-principles--documentation">Principles</a>.

---

## Which container for Create or Update?

```
Start
 │
 ├─ Does the flow branch on early answers?
 │   (different steps for different choices)          ── yes ──► Wizard
 │
 ├─ Are there validation gates that must pass
 │   before later steps can be filled?                ── yes ──► Wizard
 │
 ├─ Does the user need to see the collection
 │   while editing (compare, edit sequentially)?      ── yes ──► Side panel
 │
 ├─ Does the item have multiple sub-views,
 │   deep linking, or >~30 fields?                    ── yes ──► Dedicated page
 │
 ├─ Is the edit a single field with trivial
 │   validation, used in flow across many items?      ── yes ──► Inline edit
 │
 └─ Otherwise                                         ─────────► Centered overlay
```

## Which surface for Read?

```
Start
 │
 ├─ Does the URL need to be shareable / bookmarkable? ── yes ──► Dedicated page
 │
 ├─ Is the content deep, multi-section, or tabbed?    ── yes ──► Dedicated page
 │
 ├─ Are most reads shallow but some need depth?       ── yes ──► Detail-then-escalate
 │     (light surface + "Open full view")
 │
 ├─ Does the user benefit from background context?    ── yes ──► Side panel
 │
 └─ Otherwise                                         ─────────► Centered overlay
```

## Where does this action go?

```
Question: "Who is the subject of this action?"

  The collection itself ──────► Collection-level
    ├─ Single dominant intent   ► Primary
    ├─ Supporting alternative   ► Secondary (expanded if frequent)
    └─ Rare / destructive       ► Overflow

  One specific item ──────────► Item-level
    ├─ Most-likely-next         ► Primary tier
    ├─ Supporting               ► Secondary tier
    └─ Rare / destructive       ► Overflow tier

  A user-defined subset ──────► Bulk
    ├─ Common bulk operation    ► Bulk primary (in bar)
    └─ Less common / dangerous  ► Bulk overflow (in bar menu)
```

If you cannot place an action with this tree, the action is poorly defined.
Sharpen the verb.

## Should this action be critical?

Check any that apply. If **any** box is checked, treat the action as critical:

- The change cannot be reversed within the product
- The change cannot be reversed within a reasonable time window even with admin help
- The change deletes or unlinks user-visible content
- The change has side effects on users other than the one performing it
- The change triggers external side effects (emails, payments, integrations)
- The change affects compliance, audit, or legal posture
- The action's name contains "delete", "remove", "revoke", "cancel", or "destroy"

**Critical actions should:**

- Default to overflow — surfacing outside overflow is acceptable when the context
  and resource justify it (e.g., deletion is a core frequent workflow)
- Always require a confirmation overlay (regardless of placement)
- Use destructive styling on the affordance itself if surfaced, and always at the
  confirmation step

## Async feedback location

```
Work duration < 1s ──────────────► Action-level only (button spinner)
Work duration 1-5s ──────────────► Action-level + disable submit
Work duration > 5s, single item ─► Container-level (block input, show progress)
Bulk operation ──────────────────► Bar-level (aggregate progress, per-item errors)
Background work, no user waiting ► Toast / notification (the only case where
                                    global feedback is acceptable)
```
