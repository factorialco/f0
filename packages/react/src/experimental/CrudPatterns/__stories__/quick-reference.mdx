# Quick Reference

A single-page cheat sheet for CRUD pattern decisions. No prose — just the tables
and trees. For rationale, see the linked pages.

---

## Action placement

```
Who is the subject?

  Collection ────────► Collection-level
    ├─ Dominant intent  ► Primary (one button, top-right)
    ├─ Supporting       ► Secondary (max 2 expanded + overflow)
    └─ Rare/destructive ► Overflow (ellipsis)

  One item ──────────► Item-level
    ├─ Most-likely-next ► Primary tier
    ├─ Supporting       ► Secondary tier
    └─ Rare/destructive ► Overflow tier (always)

  Selection ─────────► Bulk
    ├─ Common           ► Bulk bar primary (max 2)
    └─ Rare/destructive ► Bulk bar overflow
```

---

## Container selection (Create / Update)

```
Branching or validation gates?          ► Wizard
Need to see collection while editing?   ► Side panel
Deep resource, 30+ fields, tabs?        ► Dedicated page
Single field, trivial, repetitive?      ► Inline edit
Otherwise?                              ► Centered overlay (default)
```

---

## Container selection (Read)

```
URL must be shareable / deep-linkable?  ► Dedicated page
Multi-section, tabbed, deep?            ► Dedicated page
Most reads shallow, some need depth?    ► Detail-then-escalate
Need background context?                ► Side panel
Otherwise?                              ► Centered overlay (default)
```

## Async feedback location

```
< 1s              ► Button spinner only
1–5s              ► Button spinner + disable submit
> 5s, single item ► Container-level (block + progress)
Bulk              ► Bar-level (aggregate + per-item errors)
Background        ► Toast (only acceptable global feedback case)
```

---

## Critical action checklist

An action is **critical** if ANY of these are true:

- Cannot be reversed in-product
- Deletes or unlinks user-visible content
- Side effects on other users
- Triggers external effects (emails, payments, integrations)
- Affects compliance/audit/legal
- Name contains: delete, remove, revoke, cancel, destroy

**Critical actions should:** default to overflow (surfacing is acceptable when
context justifies it), always require confirmation, use destructive styling on
the affordance if surfaced and at the confirmation step.
