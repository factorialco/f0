# Bulk Operations & Async Feedback

Bulk operations and async feedback are tightly coupled: most bulk actions take
time, and users need to understand what is happening, to what, and whether it
worked.

---

## Bulk is a different beast from single

A bulk action is not "a single action repeated N times". The differences:

| Aspect | Single | Bulk |
|---|---|---|
| Scope clarity | Obvious (one item) | Ambiguous (page? all results?) |
| Failure mode | Binary (success/fail) | Partial (some succeed, some fail) |
| Confirmation need | Depends on severity | Always, regardless of severity |
| Duration | Usually instant | Often async |
| Feedback granularity | One result | Aggregate + per-item detail |

---

## Selection scope must be explicit

The most dangerous bulk UX failure: the user thinks they selected everything but
only affected the visible page.

**Rule:** whenever the dataset can exceed one page, selection scope must be made
explicit through a **selection-scope banner**.

Banner pattern:
> "25 items on this page are selected. **Select all 1,432 items matching your filters.**"

This banner appears after the user checks the "select all on page" affordance,
and offers to expand to the full result set. The expansion requires an explicit,
deliberate click.

## Server-side selection (predicate selection)

For large datasets, representing selection as "an array of IDs" breaks down. The
correct abstraction is a **predicate**: "all items matching the current filter."

Bulk action handlers must accept either:
- A list of specific item identifiers, OR
- A predicate (the active filter state + "all selected" flag)

This distinction is invisible to the user but critical for implementation. It
enables bulk actions on millions of records without enumerating them client-side.

## Bulk action bar

The floating bar that appears when items are selected:

**Anatomy:**
1. **Counter** — "N selected" (must reflect actual scope: page or all)
2. **Primary actions** — up to two buttons (the most common bulk operations)
3. **Overflow** — additional or destructive bulk actions
4. **Cancel** — clears selection, always visible, always immediate

**Behavior:**
- Appears when one or more items are selected
- Floats at the bottom (or top) of the viewport, sticky
- Disappears when selection is cleared or action completes (unless keepSelection is desired)

## Async feedback

Most bulk actions — and some single-item actions — are not instant. Users need
to know: is it working? did it finish? did anything fail?

### Feedback state machine

Every async action moves through four states:

```
idle → loading → success
                → error
```

- **idle**: affordance is interactive, ready to be triggered.
- **loading**: affordance shows a spinner, further clicks are disabled.
- **success**: brief confirmation (checkmark, color change), then return to idle.
- **error**: error state with message, retry option, then return to idle.

### Controlled vs. auto-managed

Two modes for managing the state machine:

- **Auto-managed**: the system moves through states automatically based on the
  promise resolution of the action handler. Simpler, covers 80% of cases.
- **Controlled**: the consuming code explicitly sets the state. Required when the
  action opens a confirmation dialog (the state must remain idle until the user
  confirms) or when multi-step server communication is involved.

## Partial failure in bulk

Partial success is the common case for bulk operations. Design for it:

- **Report aggregate**: "11 of 14 succeeded. 3 failed."
- **Surface per-item errors**: let users see which items failed and why.
- **Offer retry**: "Retry 3 failed items" as a follow-up action.
- **Don't clear selection on partial failure**: keep the failed items selected so
  the user can retry or inspect them.
