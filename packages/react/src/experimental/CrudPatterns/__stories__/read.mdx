# Read

Read is the most frequent CRUD operation and should be the **cheapest** in terms
of user effort. The goal: let users glance at an item's details and return to the
collection with minimal friction.

---

## Choose the right default for the resource

Not every read starts with an overlay. The right default depends on the resource:

- **Simple resources** (few fields, no sub-views): a dismissible overlay or side
  panel is the cheapest path. The user glances and returns.
- **Deep resources** (multi-tab, shareable URL, nested content): navigating directly
  to a dedicated page is appropriate. Forcing an intermediate overlay adds friction
  without value.

The goal is to match the read container to the resource's complexity — not to
always minimize the container weight.

## When to escalate to a page

A dedicated page is appropriate for read when:

- The item has **multiple sub-views** (tabs, activity feeds, file lists, permissions)
- The item is **shared via URL** — deep-linkability is a requirement
- The item has **so much content** that an overlay cannot hold it comfortably
- Further actions branch from the read view into their own multi-step flows

If only some of these apply some of the time, consider the **detail-then-escalate**
pattern (see [Containers](/?path=/docs/experimental-crud-patterns-containers--documentation)).

---

## Detail-then-escalate

A two-layer approach:

1. **Light surface** (overlay or side panel): shows a summary — key fields, status,
   recent activity. Exposes one prominent affordance: "Open full view" or
   "See details".
2. **Heavy surface** (dedicated page): the full resource with all tabs, history,
   and nested content.

The light surface is sufficient for ~80% of reads. The remaining 20% escalate
deliberately.

## Read on different surface densities

The trigger for read and the container it opens into depend on the surface:

**Dense surfaces (tabular data):**

- Item click or row click opens a read overlay or navigates to a page.
- Hover-revealed actions may include an explicit "View" action, but clicking the
  row itself is the primary read affordance.
- Exception: editable surfaces should _not_ use row-click for read — it conflicts
  with cell focus for editing.

**Sparse surfaces (cards, list items):**

- Item click opens a read overlay.
- The card/item body itself is the read affordance.
- Edit and other actions are exposed _inside_ the read view, not on the card surface.

**Board/kanban surfaces:**

- Card click opens a read overlay (typically centered).
- The overlay serves as a hub: read the item, then edit, delete, or reassign from within.

## Read as a gateway to other operations

The read view often serves as the **launching point** for Update and Delete:

- Edit action exposed in the read view's header or footer
- Critical actions (delete, archive) in the read view's overflow menu

This "read-first" pattern is especially important for content-rich items where
the user needs context before deciding to mutate. See
[Create & Update](/?path=/docs/experimental-crud-patterns-create-update--documentation)
for more on this pattern.
