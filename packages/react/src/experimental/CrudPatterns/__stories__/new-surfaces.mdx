# Applying CRUD Patterns to New Surfaces

When designing a CRUD experience for a surface type that doesn't exist yet —
timeline, map, graph, calendar, tree, or anything novel — use this page as a
framework for making decisions from first principles.

---

## Step 1: Characterize the surface

Answer these questions about the new surface:

## Step 2: Map the action hierarchy

Regardless of surface type, the hierarchy stays the same:

1. **Collection-level primary** — identify the single dominant intent. Place it in
   the surface's "frame" (toolbar, header, floating button, sidebar — wherever the
   surface's chrome lives).

2. **Item-level actions** — decide the reveal strategy based on density:
   - High density (many items visible): hover or long-press to reveal
   - Low density (few items, large): persistent affordances on/near the item
   - Occluding surfaces (map markers, overlapping nodes): actions in a popover on select

3. **Bulk actions** — decide how selection works:
   - If items are discrete and countable: checkbox selection + bar
   - If items are spatial (regions, ranges): lasso/range selection + bar
   - If selection is hard to visualize: consider whether bulk is appropriate at all

## Step 3: Choose containers

The container decision is independent of the surface. The same rules apply:

- Centered overlay for self-contained operations
- Side panel when the surface context matters during editing
- Dedicated page for deep resources
- Inline edit only if the surface supports in-place focus without ambiguity

**Special considerations for novel surfaces:**
- If the surface is spatial (map, graph), side panels often work better than
  centered overlays because the user wants to see how their edit affects the layout.
- If the surface is temporal (timeline, calendar), confirm that "Create" has a
  clear temporal anchor (which date? which slot?) — scoped creation is almost
  always appropriate here.

## Step 4: Define the Read trigger

Every surface needs a way to open a read view. Decide:

<Unstyled>
  <table className="mb-8 w-full dark:text-f1-foreground-inverse/80">
    <thead>
      <tr>
        <th className="text-left">Surface characteristic</th>
        <th className="text-left">Read trigger</th>
      </tr>
    </thead>
    <tbody>
      <tr>
        <td>Items are clickable regions (cards, nodes, markers)</td>
        <td>Click/tap the item</td>
      </tr>
      <tr>
        <td>Items are rows/lines (table, list, timeline entries)</td>
        <td>Click the row</td>
      </tr>
      <tr>
        <td>Items overlap or are small (map pins, dense graph nodes)</td>
        <td>Click to select → popover with "Open"</td>
      </tr>
      <tr>
        <td>Items are editable in-place (cells, inline fields)</td>
        <td>Explicit "View" action (not click — that's edit)</td>
      </tr>
    </tbody>
  </table>
</Unstyled>

The key conflict to avoid: **click-to-read vs. click-to-edit**. If the surface is
editable, these two gestures must be disambiguated. The safest solution is an
explicit "View details" item action.

---

## Step 5: Handle destructive actions

Destructive patterns are surface-agnostic. Regardless of how novel the surface is:

- Delete lives in overflow (always)
- Confirmation names the target
- Bulk delete requires visible scope

The only surface-specific decision: **where does the overflow menu live?**
- On item hover/select: popover or context menu
- On a detail view: header overflow
- In bulk: the bar's overflow section
