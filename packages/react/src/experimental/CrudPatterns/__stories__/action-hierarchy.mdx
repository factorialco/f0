# Action hierarchy

Actions are organized in a strict hierarchy based on **who or what the action
operates on** and **how often it is used**. This hierarchy determines placement,
visibility, and affordance strength.

## Collection-level actions

### Primary action

The single, dominant call-to-action mounted on the collection's frame. Almost
always "Create". There is only one.

- Placed in the most prominent position (typically top-right of the collection header).
- If multiple creation flows exist (e.g., "Create manually" vs. "Import from file"),
  collapse into a split button: one default action + a chevron revealing alternatives.
- Label must be a verb: "Add employee", "Create report" — never just a noun.

### Secondary actions

Supporting operations on the collection: import, export, share, refresh, configure.

- Cap visible (expanded) secondary actions at **two**. Anything beyond goes into overflow.
- Secondary actions can optionally hide their label and show icon-only when expanded
  (useful for density).
- Group related actions with separators in the overflow menu.

### Overflow ("More options")

Rarely-used collection actions and anything that doesn't warrant a permanent slot.

- Always represented by an ellipsis icon trigger.
- Critical collection actions (e.g., "Delete all") live here, never in primary or secondary.

Actions that operate on a single item. They have three internal tiers that
control how they render depending on the visualization's density:

### Conditional item actions

Not every item supports every action. Item actions can be defined as a function
of the item itself — returning different sets (or nothing at all) depending on
the item's state, permissions, or type.

### Navigation as an action

Clicking an item to navigate to its detail view is itself an action. It can be
expressed as a URL (enabling right-click, open in new tab) or as a click handler.
This "implicit primary action" is separate from the explicit item actions menu.

## Bulk actions

Actions that operate on a user-defined selection of items. They surface in a
**floating action bar** that appears when one or more items are selected.

### Bulk action bar anatomy

1. **Counter** — "N selected" reflecting current scope (page or all results).
2. **Primary bulk actions** — up to two buttons in the bar.
3. **Secondary bulk actions** — overflow menu in the bar.
4. **Cancel / clear** — always present, always immediate.

### Selection-scope banner

When the dataset spans multiple pages, a banner clarifies the scope:
"You selected 25 items on this page. **Select all 1,432 matching items.**"

This prevents the most dangerous bulk UX failure: the user thinks they selected
everything but only affected one page.

## Placement decision tree

When placing a new action, ask: **"Who is the subject?"**

```
The collection itself ──────► Collection-level
  ├─ Single dominant intent  ► Primary
  ├─ Supporting alternative  ► Secondary (expanded if frequent)
  └─ Rare / destructive      ► Overflow

One specific item ──────────► Item-level
  ├─ Most-likely-next        ► Primary tier
  ├─ Supporting              ► Secondary tier
  └─ Rare / destructive      ► Overflow tier

A user-defined subset ──────► Bulk
  ├─ Common bulk operation   ► Bulk primary (in bar)
  └─ Less common / dangerous ► Bulk overflow (in bar menu)
```

If you cannot place an action with this tree, the action is poorly defined.
Sharpen the verb.
