# CRUD Patterns — Agnostic Reference

A product-agnostic vocabulary, a set of principles, and a decision toolkit for
designing **Create / Read / Update / Delete** experiences in any surface, any
framework, any platform.

This documentation is intentionally free of component names, framework names, and
implementation specifics. It describes how CRUD _thinks_, not how it ships.

## How to read this documentation

It is **layered**. Each part stands on the part before it.

1. **<a target="_parent" href="/?path=/docs/experimental-crud-patterns-principles--documentation">Principles</a>** — the foundational rules. The "why".
2. **<a target="_parent" href="/?path=/docs/experimental-crud-patterns-action-hierarchy--documentation">Action hierarchy</a>** — how to classify and place actions.
3. **<a target="_parent" href="/?path=/docs/experimental-crud-patterns-containers--documentation">Containers</a>** — the surfaces that host CRUD operations.
4. **<a target="_parent" href="/?path=/docs/experimental-crud-patterns-create-update--documentation">Create & Update</a>** — patterns for making and changing things.
5. **<a target="_parent" href="/?path=/docs/experimental-crud-patterns-read--documentation">Read</a>** — patterns for viewing things.
6. **<a target="_parent" href="/?path=/docs/experimental-crud-patterns-delete-destructive--documentation">Delete & destructive</a>** — patterns for removing things safely.
7. **<a target="_parent" href="/?path=/docs/experimental-crud-patterns-bulk-async--documentation">Bulk & async</a>** — patterns for operating at scale.
8. **<a target="_parent" href="/?path=/docs/experimental-crud-patterns-decisions--documentation">Decisions</a>** — heuristics for picking between patterns.

If you disagree with a decision in section 8, the dispute will almost always trace
back to a principle in section 1. Resolve it there.

## Scope

These patterns apply whenever a user interacts with a **collection of records** —
regardless of how those records are visualized (tables, lists, cards, boards,
editable grids, or anything else).

The patterns are concerned with:

- **Where** an action lives (which surface, which tier of hierarchy)
- **How** it is triggered (click, selection, inline focus)
- **What container** hosts the resulting task (overlay, panel, page, inline)
- **What feedback** the user receives (sync, async, success, failure)

They are _not_ concerned with specific field layouts, validation logic, or visual
styling — those are implementation details.

        <td>Affordance</td>
        <td>
          A perceptible cue that an action is possible. A button is an
          affordance; a row that becomes clickable on hover is a weaker one.
        </td>
      </tr>
      <tr>
        <td>Surface</td>
        <td>
          A region of the interface that hosts actions and content (a page, a
          panel, a row, a dialog).
        </td>
      </tr>
      <tr>
        <td>Container</td>
        <td>A surface that opens to host a task and closes when done.</td>
      </tr>
      <tr>
        <td>Scope</td>
        <td>
          The set of items an action will affect. Always make it explicit before
          the user commits.
        </td>
      </tr>
      <tr>
        <td>Critical action</td>
        <td>
          An action that is irreversible, destructive, or has side effects on
          other users.
        </td>
      </tr>
      <tr>
        <td>Predicate selection</td>
        <td>
          A way to represent "all items matching X" without enumerating them.
          Required at scale.
        </td>
      </tr>
