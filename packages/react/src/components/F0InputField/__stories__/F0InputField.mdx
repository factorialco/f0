# F0InputField

`F0InputField` is the shared chrome primitive that every writable F0 input is composed on top of. It owns the label, status, icon, append, clear and loading affordances, plus the focus, hover, disabled, read-only and accessibility behaviour the design system guarantees across the input family.

This page is for **design-system contributors** building a new type of input (for example `F0PercentageInput`, `F0CurrencyInput`, `F0PhoneInput`). In product code, always reach for the dedicated `F0*Input` component documented under the [Inputs/](?path=/docs/components-inputs-text-input--documentation) group.

---

## Overview

### Anatomy

<Canvas of={Stories.Default} />

The `F0InputField` is composed of several parts:

- **Label**: text describing the input's purpose. Always rendered into the DOM (use `hideLabel` to keep it screen-reader only).
- **Input area**: where the cloned `children` element is rendered. Fills the remaining width.
- **Icon**: optional, renders **before** the input area.
- **Append / appendTag / buttonToggle**: optional, render **after** the input area.
- **Clear button**: optional (`clearable={true}`), rendered only while the value is non-empty.
- **Messages**: status (`error` / `warning` / `info`) or `hint` below the field. `error` wins.
- **Character count**: rendered to the right of the label when `maxLength` is set, unless `hideMaxLength`.

## Guidelines

### Design best practices

#### When to use

- You are building a **new writable input** for the design system (e.g. `F0PercentageInput`,
  `F0CurrencyInput`, `F0PhoneInput`, `F0MaskedInput`).
- You need the standard F0 input chrome around a non-standard editable surface (custom
  `contenteditable`, masked input, third-party widget that exposes `value`/`onChange`).

#### When NOT to use

- You are writing **product code** and need a text, number, search, textarea or duration input.
  Use the dedicated component under [Inputs/](?path=/docs/components-inputs-text-input--documentation)
  instead.
- You only need a button or a non-editable visual chip. `F0InputField` always wraps an editable
  element; use `F0Button`, `F0Chip` or `F0Tag` for non-editable surfaces.

### API contract

`F0InputField` is a **chrome contract**, not a visual one. When composing a new writable input on top of it:

**Consumer must provide**

- A single editable React element as `children` (`<input>`, `<textarea>`, `contenteditable`, third-party widget exposing `value`/`onChange`/`onFocus`/`onBlur`/`disabled`/`readonly`). F0InputField clones it and injects `value`, `id`, sizing className and aria attributes.
- A controlled `value` + `onChange` pair. F0InputField does not own state.
- A `label` (use `hideLabel` if the design omits it visually — still required for screen readers).
- Type-specific behaviour: parsing, masking, debounce, validation, formatting.

**F0InputField does NOT**

- Render its own `<input>`. Without `children` it renders empty chrome.
- Manage focus programmatically. Forward `ref` from the consumer to the editable element.
- Parse, validate or format values.
- Set `aria-*` attributes on the editable element manually — the wrapper injects them via `cloneElement`.

## Behavior

What `F0InputField` guarantees when the [API contract](#api-contract) is satisfied.

A11y wiring is not visually demonstrable. It includes: `aria-busy`, `aria-disabled`, `aria-controls`, `aria-expanded`, label association via `id`, `aria-live`-polite status region, autofill detection, and `aria-label="Clear"` on the clear button (override via `onClear`).

### Status precedence

When both `hint` and `status={{ type: "error", ... }}` are provided, the error message replaces the hint and the critical border colour wins. The legacy boolean `error` prop behaves the same way.

<Canvas of={Stories.StatusPrecedence} />

### Composition with textarea

The cloned `children` element can be a `<textarea>` instead of an `<input>`. Sizing, status and autofill detection work identically; the chrome adapts to the multi-line surface.

<Canvas of={Stories.WithATextarea} />

### Character counter

The example below sets `maxLength={10}` and a `value` of `"long text should be truncated"` (29 characters). On mount, `F0InputField` substrings the value to 10 characters; the counter renders `10/10` next to the label.

<Canvas of={Stories.WithMaxLength} />

### Autofill

Browsers do not fire `change` events on autofill, which would leave the floating placeholder visible on top of the autofilled value. `F0InputField` listens to the WebKit `animationstart` event and polls `:-webkit-autofill` / `:autofill` to hide the placeholder until the autofill state clears.
