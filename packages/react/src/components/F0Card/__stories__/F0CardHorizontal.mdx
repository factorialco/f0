# Card Horizontal

A horizontal card for items shown as a **vertical stack of standalone cards** — an
optional avatar, a title and an optional description on the left, actions on the
right. Each item keeps its own surface and spacing (it is not a table row or a
divided list); for a vertical, content-rich card with an image, metadata or
selection, use `F0Card`.

## Guidelines

### Design best practices

`F0CardHorizontal` and `F0Card` overlap — both can carry an avatar, a title, a
description and actions — so the same item could often be built with either. The
deciding question is **the user's job, not the parts**:

- Reach for `F0CardHorizontal` when the item is **one scannable card in a vertical
  stack** and the user works down many of them, acting on each — open,
  approve/reject, toggle, select.
- Reach for `F0Card` when the item is a **standalone object the user reads or
  explores** — media, several metadata facts, multiple sections — usually a tile in a
  grid or gallery.

When an item could go either way, default to `F0CardHorizontal` if it lives in a list
and the task is scanning or acting; switch to `F0Card` once it needs an image, more
than a couple of metadata facts, or a grid/tile layout.

### Content best practices

- **Title**: the primary line. Keep it to one scannable phrase — a name, a subject, a question
- **Description**: optional secondary line. Keep it short; it wraps when long and grows the card
- **Action labels**: verb-first, no articles
  - **Correct:** "Approve", "Open", "Renew"
  - **Incorrect:** "Approval", "Open the card", "Click to renew"
- **Status label**: required — the icon carries the meaning, so name the outcome ("Accepted", "Rejected") for assistive tech

### Behavior

- The card becomes a click target **only** when `link` or `onClick` is set (opt-in): it then gains a pointer cursor and a hover/focus affordance. Otherwise only its actions are interactive — the card is driven by its buttons or by a whole-card click, never both.
- `stackAt` reacts to the **container** width (a container query on the card itself), not the viewport — so the actions stack correctly inside grids, columns and drawers. Below the breakpoint they drop to their own line with a separator, and secondary buttons fold into the left ⋯ menu.
- `status` reports a resolved outcome and **replaces** the actions; `confirmAction` / `rejectAction` render the icon-only approve/reject pair. Both take precedence over the standard actions.
- The avatar renders at a fixed `lg` size on the left; types without an `lg` variant render at their intrinsic size.
- `inactive` is purely presentational — it strikes through and dims the text (e.g. a voided card). Pair it with the matching `status`.

## Accessibility

- The `status` icon conveys the outcome through its required `label` — assistive technology reads the label, so never rely on the icon or colour alone.
- A whole-card `link` renders a full-card overlay link labelled by the title; action buttons stop click propagation, so they stay operable independently of the card link.
- Actions and the overlay link are reachable with `Tab` and activated with `Enter`.
