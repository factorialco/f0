# Card

Displays a single entity — a person, document, project, or job opening — as a self-contained, optionally clickable surface that groups an avatar, title, description, metadata, and actions. Use it in grids and galleries where each item needs visual weight and several pieces of information. For a dense, single-row list item use `F0CardRow`; for a plain styled container of arbitrary content use `F0Box`.

### Design best practices

**When to use**

Reach for a Card when each item is an entity that benefits from visual grouping —
an avatar, a few metadata rows, and optional actions — typically arranged in a
grid or gallery.

### Content best practices

- Title is sentence case with no trailing period; it names the entity.
- Keep the description to one short, informative line — it truncates after 3
  lines (2 when `compact`).
- Action labels are imperative and short.
  - **Correct:** "Refer", "Apply", "View more"
  - **Incorrect:** "Referral", "Application submitted", "Click to view more"

### Behavior

- **Full-card link:** when `link` is set, the entire card becomes clickable via
  a full-card overlay anchor that uses `title` as its accessible label. `onClick`
  still fires alongside navigation.
- **Interactive children stay interactive:** content inside `children` (links,
  buttons, inputs, toggles) keeps its own pointer events and stops propagation,
  so it works even when the card itself is a `link`.
- **Image overlay controls:** when an `image` is present, the selection checkbox,
  bookmark toggle and overflow menu move into an overlay on the image; otherwise
  they sit in the top-right of the header.
- **Bookmark visibility:** the bookmark toggle appears on hover and stays visible
  while `bookmarked` is `true`, so the saved state is always recognisable.
- **Controlled selection and dismiss:** `selected`/`onSelect` and the alert's
  `visible`/`onDismiss` are controlled — the consumer owns the state.

### Screen reader behavior

- When `link` is set, the card header exposes `role="button"` and the full-card
  overlay anchor uses `title` as its `aria-label` — always provide a meaningful
  `title` so the destination is announced.
- The bookmark toggle uses its `label` (falling back to `title`) as the
  accessible name, so the save control is announced even though it is icon-only.
- The skeleton state sets `aria-busy="true"` and `aria-live="polite"` so loading
  is announced while data is fetched.
- Avatar images are decorative relative to the title; the title carries the
  meaning. Ensure any `image` is supported by the `title` text, since it is used
  as the image `alt`.
