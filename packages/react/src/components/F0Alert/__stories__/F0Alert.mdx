# Alert

Keeps users aware of something relevant to their current context — a missing
setup, a failed process, a completed action — without interrupting the workflow.
For transient feedback use a toast; for decisions that block the workflow use
`F0Dialog`.

### Content best practices

- **Title**: Sentence case, no period. State the situation clearly ("Company page inactive")
- **Description**: One or two sentences. Explain what the situation means and what can be done about it
- **Action label**: Verb-first, no articles ("Request info", "Activate now")
- **Link label**: Descriptive of the destination ("See all invoices"), never generic ("Click here")

### Behavior

- The alert is persistent by default — it does not dismiss automatically. It remains visible until the underlying condition changes or the parent component removes it. This is what distinguishes it from a toast.
- `onClose` is optional. Only provide it when the product team decides users should be able to dismiss the alert. When omitted, no close button is rendered and the alert cannot be dismissed.
- When `onClose` is provided, the parent controls visibility and reappearance — for example, hiding the alert on close and showing it again after a timeout or on next page load if the issue is still unresolved.
- At narrow container widths the layout stacks vertically: icon and text on top, action and link below
- At wider containers the layout is horizontal: icon and text on the left, action and link on the right
- The alert stretches to fill its container — control width from the parent
- When `action.disabled` is true, the button remains visible but non-interactive — keep it visible so the action remains discoverable

## Accessibility

- The component renders as a plain `<div>` with no ARIA role. It is not announced automatically when it appears — place it in the page on initial render or ensure the surrounding context gives it enough prominence.
- For alerts that appear dynamically in response to an action (e.g. after a failed save), wrap the alert in an `aria-live="polite"` region so screen readers announce it without interrupting the current flow.
- The icon (rendered via `F0AvatarAlert` or `F0AvatarIcon`) is decorative — the title and description carry the full meaning. Do not rely on the icon alone to communicate severity.
- The `action` button and `link` are fully keyboard-accessible via `Tab` and `Enter`.

## Usage

### Narrow layout

At small container widths the layout stacks vertically — useful for sidebars,
panels, or any tight container.

<Canvas of={Stories.Narrow} />

```tsx
<F0Alert
  variant="warning"
  title="Company page inactive"
  description="You are missing the company page. It is mandatory to be activated."
  action={{ label: "Request info", onClick: () => {} }}
/>
```

### Disabled action

Use when the action is conditionally unavailable. Keep the button visible so
the action is discoverable even when blocked.

<Canvas of={Stories.DeactivatedAction} />

```tsx
<F0Alert
  variant="warning"
  title="Company page inactive"
  description="You are missing the company page. It is mandatory to be activated."
  action={{ label: "Request info", onClick: () => {}, disabled: true }}
/>
```

### Dismissible alert

Pass `onClose` to show a close button. The parent controls visibility and
reappearance — hide the alert on close and decide when it should come back
based on product logic (e.g. timeout, next session, or once the issue is resolved).

<Canvas of={Stories.Dismissible} />

```tsx
import { useState } from "react";

const [visible, setVisible] = useState(true);

return visible ? (
  <F0Alert
    variant="info"
    title="Your workspace includes up to 3 invoices."
    description="Start creating invoices: it's free with your plan!"
    action={{ label: "Request info", onClick: () => {} }}
    onClose={() => setVisible(false)}
  />
) : null;
```
