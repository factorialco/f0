# Guidelines

When to reach for cocreation, when to use something else, and the principles that keep the
human in control of what the AI produces.

## When to use

<Unstyled>
  <table className="mb-8 w-full dark:text-f1-foreground-inverse/80">
    <thead>
      <tr>
        <th className="text-left">Situation</th>
        <th className="text-left">Use cocreation when</th>
      </tr>
    </thead>
    <tbody>
      <tr>
        <td>Creating a structured resource from a vague intent</td>
        <td>
          The user can describe the goal in words but a blank form would be slow
          or intimidating (a survey, a job description, a policy draft).
        </td>
      </tr>
      <tr>
        <td>The first draft benefits from a conversation</td>
        <td>
          Clarifying questions meaningfully improve the output before generation
          (audience, tone, length, scope).
        </td>
      </tr>
      <tr>
        <td>The resource is refined iteratively</td>
        <td>
          The user wants to keep adjusting the draft while staying in dialogue,
          rather than committing once and editing later.
        </td>
      </tr>
      <tr>
        <td>Context from the originating surface matters</td>
        <td>
          Creation should happen over the collection or page the user came from,
          not in a separate tool.
        </td>
      </tr>
    </tbody>
  </table>
</Unstyled>

## Welcome Screen vs Guided Chat

A cocreation flow can open the conversation two ways. Both land in the same Chat → Split
machinery; they differ only in how the first turn is framed. See both running in the
[Walkthrough](?path=/docs/patterns-ai-cocreation-walkthrough--documentation).

<Unstyled>
  <table className="mb-8 w-full dark:text-f1-foreground-inverse/80">
    <thead>
      <tr>
        <th className="text-left">Entry</th>
        <th className="text-left">What the user sees first</th>
        <th className="text-left">Reach for it when</th>
      </tr>
    </thead>
    <tbody>
      <tr>
        <td>
          <strong>Welcome Screen</strong>
        </td>
        <td>
          A full-width welcome screen — a grid of entry-point cards below the
          composer and starter-prompt suggestions above it. The user chooses a
          direction.
        </td>
        <td>
          There are several equally likely starting points and no single obvious
          first move, so the value is in showing the menu.
        </td>
      </tr>
      <tr>
        <td>
          <strong>Guided Chat</strong>
        </td>
        <td>
          An opening line posted on the user's behalf and one clarifying
          question — triaging the resource type, or the entry action — with the
          composer hidden until it's answered.
        </td>
        <td>
          There is one clear first action. The flow already knows what to ask,
          so it asks it.
        </td>
      </tr>
    </tbody>
  </table>
</Unstyled>

**Prefer Guided Chat when there is one clear initial action.** A started conversation beats a
blank canvas: a single question the user just answers is lower-cost than a screen of options they
have to read and choose between. Reserve the Welcome Screen for flows where the starting points are
genuinely varied and worth presenting as a menu.

## Principles

<DoDonts
  do={{
    description: "Keep the human in the loop — confirm before applying.",
    guidelines: [
      "Confirm the first draft too, not only later edits — the initial generation is an accept / reject card as well",
      "Surface every AI-proposed change as an explicit accept / reject card",
      "Let the user edit the draft directly, not only through the chat",
    ],
  }}
  dont={{
    description: "Don't let the AI write content into the resource silently.",
    guidelines: [
      "Don't draft onto the canvas before the user has accepted the draft",
      "Don't auto-apply proposals the user has not seen",
      "Don't auto-commit destructive changes",
    ],
  }}
/>

<DoDonts
