# Guidelines

When to reach for co-creation, when to use something else, and the principles that keep the
human in control of what the AI produces.

## When to use

<Unstyled>
  <table className="mb-8 w-full dark:text-f1-foreground-inverse/80">
    <thead>
      <tr>
        <th className="text-left">Situation</th>
        <th className="text-left">Use co-creation when</th>
      </tr>
    </thead>
    <tbody>
      <tr>
        <td>Creating a structured resource from a vague intent</td>
        <td>
          The user can describe the goal in words but a blank form would be slow or
          intimidating (a survey, a job description, a policy draft).
        </td>
      </tr>
      <tr>
        <td>The first draft benefits from a conversation</td>
        <td>
          Clarifying questions meaningfully improve the output before generation (audience,
          tone, length, scope).
        </td>
      </tr>
      <tr>
        <td>The resource is refined iteratively</td>
        <td>
          The user wants to keep adjusting the draft while staying in dialogue, rather than
          committing once and editing later.
        </td>
      </tr>
      <tr>
        <td>Context from the originating surface matters</td>
        <td>
          Creation should happen over the collection or page the user came from, not in a
          separate tool.
        </td>
      </tr>
    </tbody>
  </table>
</Unstyled>

## Principles

<DoDonts
  do={{
    description: "Keep the human in the loop — confirm before applying.",
    guidelines: [
      "Surface every AI-proposed change as an explicit accept / reject card",
      "Let the user edit the draft directly, not only through the chat",
    ],
  }}
  dont={{
    description: "Don't let the AI write changes into the resource silently.",
    guidelines: [
      "Don't auto-apply proposals the user has not seen",
      "Don't auto-commit destructive changes",
    ],
  }}
/>

<DoDonts
  do={{
    description:
      "Keep the resource visible and editable alongside the conversation during Split.",
  }}
  dont={{
    description:
      "Don't reduce the user to a reviewer who can only approve or reject opaque output.",
  }}
/>

<DoDonts
  do={{
    description:
      "Ask clarifying questions before drafting when they materially improve the result.",
  }}
  dont={{
    description:
      "Don't interrogate the user with questions the system could reasonably infer or defer.",
  }}
/>

<DoDonts
  do={{
    description:
      "Make AI changes reviewable and reversible before they are saved.",
  }}
  dont={{
    description:
      "Don't make the user start over to undo a single proposed change.",
  }}
/>

<DoDonts
  do={{
    description:
      "Match persistence to the resource — draftable resources are created up front, then saved explicitly; resources without a draft state persist only on explicit Save.",
  }}
  dont={{
    description:
      "Don't silently create records for resources that have no draft state.",
  }}
/>

<DoDonts
  do={{
    description:
      "Create and edit the resource in the canvas inside the chat, keeping the conversation alongside it.",
  }}
  dont={{
    description:
      "Don't navigate the user to a separate resource page mid-creation — it breaks the shared context.",
  }}
/>

<DoDonts
  do={{
    description:
      "Warn before navigation that would abandon an in-progress creation (a general rule for unsaved work).",
  }}
  dont={{
    description:
      "Don't discard unsaved co-creation work silently, and don't add the same warning to merely collapsing the chat panel.",
  }}
/>

<DoDonts
  do={{
    description:
      "Keep only the latest card for a resource interactive, and preserve superseded cards as history.",
  }}
  dont={{
    description:
      "Don't leave several Open/Close toggles competing for the same resource.",
  }}
/>

The [Standard flow](?path=/docs/patterns-co-creation-standard-flow--documentation) page defines
these rules in full, including the resource-card lifecycle.
