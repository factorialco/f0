# F0Form

A declarative form pattern that combines a typed Zod schema with field metadata to produce a fully functional, validated form. All field configuration — labels, placeholders, types, and validation — lives inside the schema definition rather than in JSX.

---

# Overview

## Purpose

- Eliminates the need to wire up individual field components; the form renders itself from the schema.
- Embeds labels, help text, placeholders, and validation directly in the Zod type — one source of truth per field.
- Provides consistent UI for loading, error, and success states without boilerplate.
- First-class TypeScript inference: field values are typed end-to-end from schema to submit handler.

## Architecture

F0Form is built on two pieces:

1. **`useF0FormDefinition`** — a hook that accepts the Zod schema, default values, and submit handler, and returns a `formDefinition` object.
2. **`<F0Form />`** — a render component that accepts the `formDefinition` and renders all fields automatically.

```tsx
const formSchema = z.object({
  name: f0FormField.text({ label: "Name" }),
})

const formDefinition = useF0FormDefinition({
  name: "my-form",
  schema: formSchema,
  defaultValues: { name: "" },
  onSubmit: async ({ data }) => {
    await save(data)
    return { success: true }
  },
})

return <F0Form formDefinition={formDefinition} />
```

### Enum inference from `options`

When `options` is passed to `f0FormField.select` or `f0FormField.multiSelect`, TypeScript infers a `z.enum` schema from the option values. This gives you narrowed types in the submit handler:

```tsx
role: f0FormField.select({
  label: "Role",
  options: [
    { value: "admin", label: "Admin" },
    { value: "member", label: "Member" },
    { value: "viewer", label: "Viewer" },
  ],
})
// Typed as: z.ZodEnum<["admin", "member", "viewer"]>
// data.role will be "admin" | "member" | "viewer" in the submit handler
```

When using enum-inferred select fields, use `undefined` (not `""`) as the default value for empty initial state.

### `optional` flag

All shortcuts that support `optional` accept it as a top-level config key instead of wrapping the schema manually:

```tsx
// ✓ Shortcut — optional as config flag
notes: f0FormField.textarea({ label: "Notes", optional: true })

// ✗ Equivalent — but more verbose
notes: f0FormField(z.string().optional(), {
  label: "Notes",
  fieldType: "textarea",
})
```

### `min` / `max` / `isInt` on number fields, `minLength` / `maxLength` on text fields

```tsx
age: f0FormField.number({ label: "Age", min: 0, max: 120, isInt: true })
username: f0FormField.text({ label: "Username", minLength: 2, maxLength: 20 })
```

## Using the base call (custom Zod validation)

For any validation beyond what the shortcuts provide, use the base `f0FormField(schema, config)` call with a fully custom Zod schema:

<Canvas of={Stories.Default} />

```tsx
const formSchema = z.object({
  // Shortcuts for common field types (no custom Zod needed)
  name: f0FormField.text({ label: "Name" }),
  role: f0FormField.select({
    label: "Role",
    options: [
      { value: "admin", label: "Admin" },
      { value: "member", label: "Member" },
    ],
  }),
  tags: f0FormField.multiSelect({
    label: "Tags",
    options: [
      { value: "frontend", label: "Frontend" },
      { value: "backend", label: "Backend" },
      { value: "design", label: "Design" },
    ],
  }),
  discount: f0FormField.percentage({ label: "Discount", min: 0, max: 100 }),
  budget: f0FormField.money({ label: "Budget", currency: "EUR" }),

  // Base call for custom Zod constraints not available via shortcuts
  username: f0FormField(z.string().min(2).max(20), {
    label: "Username",
    helpText: "Must be 2–20 characters",
  }),
  email: f0FormField(z.string().email("Please use a company email"), {
    label: "Email",
  }),
  age: f0FormField(
    z.number().int("Age must be a whole number").min(0).max(120),
    {
      label: "Age",
    }
  ),
})
```

The base call accepts any Zod schema (`z.string()`, `z.number()`, `z.date()`, `z.enum()`, `z.object()`, `z.array()`, etc.) and any supported field config.

## Period field

`period` is a granularity-aware period selector (Year / Half year / Quarter / Month / Range). Unlike `date` — which stores a single `Date` and re-displays it with the first granularity — a `period` field keeps the **full value** (`{ value: { from, to }, granularity }`), so the chosen granularity and range are preserved.

Use the `datePeriod` shortcut — it builds the schema for you, so there's no need to import any picker type. Label, the required asterisk, size, and error handling are wired by the form automatically.

```tsx
goalPeriod: f0FormField.datePeriod({
  label: "Goal period",
  granularities: ["year", "halfyear", "quarter", "month", "range"],
  minDate: new Date(2024, 0, 1),
  maxDate: new Date(2027, 11, 31),
  // Optional: "long" (default → "01 Aug 2025") or "default" (dd/MM/yyyy)
  displayFormat: "long",
})
```

The granularity shown when empty is the first entry in `granularities`; pre-select a period via the form's `defaultValues`. `minDate` / `maxDate` accept a static `Date` or a function of the other field values; `displayFormat` controls how the selected period reads in the input; and switching granularity in the popup only changes the view — the period is committed when a cell is picked.

# Guidelines

## Design Best Practices

### When to use

- Any create or edit form that requires validation and a submit flow.
- Multi-step wizards — use `F0WizardForm` / per-section schemas with the same `f0FormField` API.
- Inline forms inside dialogs or sidepanels.

### When NOT to use

- Simple one-field inputs with no submit action: use the standalone input component directly.
- Read-only displays: use a description list or data table instead.

### How to use

- Always embed validation rules near the field definition, not spread across other files.
- Use shortcut helpers for standard fields; reserve the base call for fields that need constraints not covered by shortcuts.
- Provide `helpText` on fields with non-obvious constraints.
- Use `row` grouping to lay out related fields side by side (e.g., first name + last name).

## Accessibility

- All fields rendered by F0Form include ARIA labels derived from the `label` config.
- Error messages are linked to their field via `aria-describedby`.
- The form's submit button manages `aria-busy` during async submission.

## Styling

The optional `styling` config controls the form layout:

- `showSectionsSidepanel` — render a sticky table-of-contents sidebar for the form's sections (see the example above). The highlighted section stays in sync with manual scrolling, not just clicks.
- `noPadding` — remove the default `p-4` padding around the form content. Use it when embedding the form flush inside a host surface (e.g. an AI canvas panel) that already provides its own spacing. Defaults to `false`, so existing forms are unaffected.

```tsx
<F0Form
  name="embedded"
  schema={schema}
  defaultValues={defaultValues}
  onSubmit={onSubmit}
  styling={{ noPadding: true }}
/>
```

## Opening a form imperatively — `forms.open`

`forms.open({ formDefinition, as })` opens a form from anywhere — no `useState` /
`useF0Form` / `formRef` wiring — and `await`s the outcome. Choose the
presentation with `as`: `"dialog"` for a single-screen form, or `"wizard"` for a
multi-step [`F0WizardForm`](?path=/docs/patterns-forms-f0wizardform--documentation).
Requires `<F0Provider>` to be mounted.

### As a dialog

Pass `mode: "dialog"` to open the form in a single-screen dialog:

```tsx
import { forms, f0FormField, useF0FormDefinition } from "@factorialco/f0-react"
import { z } from "zod"

const formDefinition = useF0FormDefinition({
  name: "add-member",
  schema: z.object({
    name: f0FormField.text({ label: "Name", minLength: 2 }),
    email: f0FormField.email({ label: "Email" }),
  }),
  defaultValues: { name: "", email: "" },
  onSubmit: async ({ data }) => {
    await save(data)
    return { success: true }
  },
})

const result = await forms.open({
  formDefinition,
  mode: "dialog",
  title: "Add team member",
  description: "They will receive an invitation email.",
})

if (result.submitted) {
  // result.data is the validated, typed form values
}
```

Behavior:

- **Submit succeeds** → the dialog closes and the promise resolves with
  `{ submitted: true, data }`.
- **Validation fails** → the dialog stays open with inline field errors.
- **Cancel / close button** → resolves with `{ submitted: false }`.

The dialog is **modal by default**, so an outside click or Escape won't dismiss
it (and discard in-progress input) — pass `modal: false` to allow that. The
dialog footer supplies the submit and cancel buttons (override their labels with
`labels: { submit, cancel }`), so you don't need `submitConfig.hideSubmitButton`
— `forms.open` sets it for you.

### As a wizard

Pass `mode: "wizard"` to open the same typed definition as a multi-step
[`F0WizardForm`](?path=/docs/patterns-forms-f0wizardform--documentation). Group
fields into steps with `sections` on the definition; `forms.open` resolves once
the **final** step submits:

```tsx
const result = await forms.open({
  formDefinition,
  mode: "wizard",
  title: "Add employee",
})

if (result.completed) {
  // result.data is the validated, typed full-form values
}
```

Behavior:

- **Final step submits successfully** → the wizard auto-closes and the promise
  resolves with `{ completed: true, data }`.
- **Dismissed / closed** → resolves with `{ completed: false }`.

If the definition uses `linkAfterLastStepSubmit`, the page navigates away and the
promise will not resolve.
