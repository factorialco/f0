# F0Form

A declarative form pattern that combines a typed Zod schema with field metadata to produce a fully functional, validated form. All field configuration — labels, placeholders, types, and validation — lives inside the schema definition rather than in JSX.

---

# Overview

## Purpose

- Eliminates the need to wire up individual field components; the form renders itself from the schema.
- Embeds labels, help text, placeholders, and validation directly in the Zod type — one source of truth per field.
- Provides consistent UI for loading, error, and success states without boilerplate.
- First-class TypeScript inference: field values are typed end-to-end from schema to submit handler.

## Architecture

F0Form is built on two pieces:

1. **`useF0FormDefinition`** — a hook that accepts the Zod schema, default values, and submit handler, and returns a `formDefinition` object.
2. **`<F0Form />`** — a render component that accepts the `formDefinition` and renders all fields automatically.

```tsx
const formSchema = z.object({
  name: f0FormField.text({ label: "Name" }),
})

const formDefinition = useF0FormDefinition({
  name: "my-form",
  schema: formSchema,
  defaultValues: { name: "" },
  onSubmit: async ({ data }) => {
    await save(data)
    return { success: true }
  },
})

return <F0Form formDefinition={formDefinition} />
```

### Enum inference from `options`

When `options` is passed to `f0FormField.select` or `f0FormField.multiSelect`, TypeScript infers a `z.enum` schema from the option values. This gives you narrowed types in the submit handler:

```tsx
role: f0FormField.select({
  label: "Role",
  options: [
    { value: "admin", label: "Admin" },
    { value: "member", label: "Member" },
    { value: "viewer", label: "Viewer" },
  ],
})
// Typed as: z.ZodEnum<["admin", "member", "viewer"]>
// data.role will be "admin" | "member" | "viewer" in the submit handler
```

When using enum-inferred select fields, use `undefined` (not `""`) as the default value for empty initial state.

### `optional` flag

All shortcuts that support `optional` accept it as a top-level config key instead of wrapping the schema manually:

```tsx
// ✓ Shortcut — optional as config flag
notes: f0FormField.textarea({ label: "Notes", optional: true })

// ✗ Equivalent — but more verbose
notes: f0FormField(z.string().optional(), {
  label: "Notes",
  fieldType: "textarea",
})
```

### `min` / `max` / `isInt` on number fields, `minLength` / `maxLength` on text fields

```tsx
age: f0FormField.number({ label: "Age", min: 0, max: 120, isInt: true })
username: f0FormField.text({ label: "Username", minLength: 2, maxLength: 20 })
```

## Using the base call (custom Zod validation)

For any validation beyond what the shortcuts provide, use the base `f0FormField(schema, config)` call with a fully custom Zod schema:

<Canvas of={Stories.Default} />

```tsx
const formSchema = z.object({
  // Shortcuts for common field types (no custom Zod needed)
  name: f0FormField.text({ label: "Name" }),
  role: f0FormField.select({
    label: "Role",
    options: [
      { value: "admin", label: "Admin" },
      { value: "member", label: "Member" },
    ],
  }),
  tags: f0FormField.multiSelect({
    label: "Tags",
    options: [
      { value: "frontend", label: "Frontend" },
      { value: "backend", label: "Backend" },
      { value: "design", label: "Design" },
    ],
  }),
  discount: f0FormField.percentage({ label: "Discount", min: 0, max: 100 }),
  budget: f0FormField.money({ label: "Budget", currency: "EUR" }),

  // Base call for custom Zod constraints not available via shortcuts
  username: f0FormField(z.string().min(2).max(20), {
    label: "Username",
    helpText: "Must be 2–20 characters",
  }),
  email: f0FormField(z.string().email("Please use a company email"), {
    label: "Email",
  }),
  age: f0FormField(
    z.number().int("Age must be a whole number").min(0).max(120),
    {
      label: "Age",
    }
  ),
})
```

The base call accepts any Zod schema (`z.string()`, `z.number()`, `z.date()`, `z.enum()`, `z.object()`, `z.array()`, etc.) and any supported field config.

# Guidelines

## Design Best Practices

### When to use

- Any create or edit form that requires validation and a submit flow.
- Multi-step wizards — use `F0WizardForm` / per-section schemas with the same `f0FormField` API.
- Inline forms inside dialogs or sidepanels.

### When NOT to use

- Simple one-field inputs with no submit action: use the standalone input component directly.
- Read-only displays: use a description list or data table instead.

### How to use

- Always embed validation rules near the field definition, not spread across other files.
- Use shortcut helpers for standard fields; reserve the base call for fields that need constraints not covered by shortcuts.
- Provide `helpText` on fields with non-obvious constraints.
- Use `row` grouping to lay out related fields side by side (e.g., first name + last name).

## Accessibility

- All fields rendered by F0Form include ARIA labels derived from the `label` config.
- Error messages are linked to their field via `aria-describedby`.
- The form's submit button manages `aria-busy` during async submission.
