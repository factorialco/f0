# F0Form

A declarative form pattern that combines a typed Zod schema with field metadata to produce a fully functional, validated form. All field configuration — labels, placeholders, types, and validation — lives inside the schema definition rather than in JSX.

---

# Overview

## Purpose

- Eliminates the need to wire up individual field components; the form renders itself from the schema.
- Embeds labels, help text, placeholders, and validation directly in the Zod type — one source of truth per field.
- Provides consistent UI for loading, error, and success states without boilerplate.
- First-class TypeScript inference: field values are typed end-to-end from schema to submit handler.

## Architecture

F0Form is built on two pieces:

1. **`useF0FormDefinition`** — a hook that accepts the Zod schema, default values, and submit handler, and returns a `formDefinition` object.
2. **`<F0Form />`** — a render component that accepts the `formDefinition` and renders all fields automatically.

```tsx
const formSchema = z.object({
  name: f0FormField.text({ label: "Name" }),
})

const formDefinition = useF0FormDefinition({
  name: "my-form",
  schema: formSchema,
  defaultValues: { name: "" },
  onSubmit: async ({ data }) => {
    await save(data)
    return { success: true }
  },
})

return <F0Form formDefinition={formDefinition} />
```

### Enum inference from `options`

When `options` is passed to `f0FormField.select` or `f0FormField.multiSelect`, TypeScript infers a `z.enum` schema from the option values. This gives you narrowed types in the submit handler:

```tsx
role: f0FormField.select({
  label: "Role",
  options: [
    { value: "admin", label: "Admin" },
    { value: "member", label: "Member" },
    { value: "viewer", label: "Viewer" },
  ],
})
// Typed as: z.ZodEnum<["admin", "member", "viewer"]>
// data.role will be "admin" | "member" | "viewer" in the submit handler
```

When using enum-inferred select fields, use `undefined` (not `""`) as the default value for empty initial state.

### `optional` flag

All shortcuts that support `optional` accept it as a top-level config key instead of wrapping the schema manually:

```tsx
// ✓ Shortcut — optional as config flag
notes: f0FormField.textarea({ label: "Notes", optional: true })

// ✗ Equivalent — but more verbose
notes: f0FormField(z.string().optional(), {
  label: "Notes",
  fieldType: "textarea",
})
```

### `min` / `max` / `isInt` on number fields, `minLength` / `maxLength` on text fields

```tsx
age: f0FormField.number({ label: "Age", min: 0, max: 120, isInt: true })
username: f0FormField.text({ label: "Username", minLength: 2, maxLength: 20 })
```

## Using the base call (custom Zod validation)

For any validation beyond what the shortcuts provide, use the base `f0FormField(schema, config)` call with a fully custom Zod schema:

<Canvas of={Stories.Default} />

```tsx
const formSchema = z.object({
  // Shortcuts for common field types (no custom Zod needed)
  name: f0FormField.text({ label: "Name" }),
  role: f0FormField.select({
    label: "Role",
    options: [
      { value: "admin", label: "Admin" },
      { value: "member", label: "Member" },
    ],
  }),
  tags: f0FormField.multiSelect({
    label: "Tags",
    options: [
      { value: "frontend", label: "Frontend" },
      { value: "backend", label: "Backend" },
      { value: "design", label: "Design" },
    ],
  }),
  discount: f0FormField.percentage({ label: "Discount", min: 0, max: 100 }),
  budget: f0FormField.money({ label: "Budget", currency: "EUR" }),

  // Base call for custom Zod constraints not available via shortcuts
  username: f0FormField(z.string().min(2).max(20), {
    label: "Username",
    helpText: "Must be 2–20 characters",
  }),
  email: f0FormField(z.string().email("Please use a company email"), {
    label: "Email",
  }),
  age: f0FormField(
    z.number().int("Age must be a whole number").min(0).max(120),
    {
      label: "Age",
    }
  ),
})
```

The base call accepts any Zod schema (`z.string()`, `z.number()`, `z.date()`, `z.enum()`, `z.object()`, `z.array()`, etc.) and any supported field config.

## Period field

`period` is a granularity-aware period selector (Year / Half year / Quarter / Month / Range). Unlike `date` — which stores a single `Date` and re-displays it with the first granularity — a `period` field keeps the **full value** (`{ value: { from, to }, granularity }`), so the chosen granularity and range are preserved.

Use the `datePeriod` shortcut — it builds the schema for you, so there's no need to import any picker type. Label, the required asterisk, size, and error handling are wired by the form automatically.

```tsx
goalPeriod: f0FormField.datePeriod({
  label: "Goal period",
  granularities: ["year", "halfyear", "quarter", "month", "range"],
  minDate: new Date(2024, 0, 1),
  maxDate: new Date(2027, 11, 31),
  // Optional: "long" (default → "01 Aug 2025") or "default" (dd/MM/yyyy)
  displayFormat: "long",
})
```

The granularity shown when empty is the first entry in `granularities`; pre-select a period via the form's `defaultValues`. `minDate` / `maxDate` accept a static `Date` or a function of the other field values; `displayFormat` controls how the selected period reads in the input; and switching granularity in the popup only changes the view — the period is committed when a cell is picked.

## Entities list field

`entitiesList` turns an **array of objects** into an editable table — one row per item, one column per item-schema field. Use it when a form needs a small, self-contained collection managed in place (links, options, team members, line items) rather than a separate page or data collection.

You define the row shape with a Zod object (or `f0FormField` shortcuts), and the columns, cell types, add/remove and reordering are derived for you. The field value is a `z.array` of that shape, validated like any other field (including `minItems` / `maxItems`).

```tsx
links: f0FormField.entitiesList({
  label: "Links",
  helpText: "Add links and drag to reorder them.",
  schema: z.object({
    title: z.string().min(1),
    url: z.string().url(),
    category: z.enum(["People", "Finance", "Legal"]),
  }),
  config: {
    labels: { addButton: "Add link" },
    maxItems: 8,
  },
})
```

### Showing only certain columns

To keep a value on each row without rendering it as a column, mark the column `hidden`. This is useful for a value that drives a row action (see below) or is only edited elsewhere:

```tsx
config: {
  columns: {
    archived: { hidden: true },
    // Per-column presentation for the visible ones:
    url: { label: "URL", placeholder: "https://…", width: 220 },
  },
}
```

Each visible column accepts `label`, `placeholder` and a fixed `width`.

### Inline vs. dialog editing

With **2 or fewer** item fields the table edits **inline** (type directly in the cells); with more it switches to a **dialog** (the add button and a per-row pencil open a form built from the same schema). Override the heuristic with `supportInlineEditing`:

```tsx
config: {
  supportInlineEditing: true
} // force inline even with many columns
```

A 2-column list edits inline — type straight into the cells:

### List-view visualization

When there's no inline editing, set `config.visualization: "list-view"` to render the items as a OneDataCollection **list** instead of a read-only table. Each row shows a title (the first field) and description lines (the rest); **Add** and the per-row **Edit** action open the form dialog, and rows have a **Remove** action. Override the row labelling with `config.listItem` (`title` / `description` / `avatar` functions). Drag-reorder isn't available in list mode.

```tsx
config: {
  visualization: "list-view",
  // optional — defaults to first field as title, the rest as description:
  listItem: {
    title: (m) => m.name,
    description: (m) => [m.email, m.role].filter(Boolean),
  },
}
```

#### Navigable rows

In list-view, add `config.itemHref` to make each row **navigate** instead of edit: the row links to the returned URL (clicking the row or its trailing arrow) and there's no edit dialog — just add and remove. This is only honored **without** an `updateFormDefinition`; when one is present the row shows the edit pencil and opens the update dialog on click instead.

```tsx
config: {
  visualization: "list-view",
  itemHref: (item) => `/resources/${item.id}`,
}
```

### Separate create / update form definitions

Instead of a single `schema` (whose add/edit dialogs share the parent form's submit), pass a `createFormDefinition` and an `updateFormDefinition` — each a `useF0FormDefinition` with its **own** `onSubmit`. Adding runs the create form's submit; editing runs the update form's submit, so the two can persist independently (e.g. `POST` vs `PATCH`). The item is still committed to the field's array value. `updateFormDefinition`'s schema is the canonical row shape (columns, display, value type), so fields only there should be `optional` to keep freshly-added rows valid.

```tsx
const createFormDefinition = useF0FormDefinition({
  schema: z.object({ name: z.string(), email: z.string().email() }),
  onSubmit: async ({ data }) => {
    await api.create(data)
    return { success: true }
  },
})
const updateFormDefinition = useF0FormDefinition({
  schema: z.object({
    name: z.string(),
    email: z.string().email(),
    role: f0FormField.select({ label: "Role", options, optional: true }),
  }),
  onSubmit: async ({ data }) => {
    await api.update(data)
    return { success: true }
  },
})

members: f0FormField.entitiesList({
  createFormDefinition,
  updateFormDefinition,
})
```

### Persisting removals

The three row operations persist through a matching trio:

| Operation | How it persists                             |
| --------- | ------------------------------------------- |
| Add       | `createFormDefinition` (its own `onSubmit`) |
| Edit      | `updateFormDefinition` (its own `onSubmit`) |
| Remove    | `config.onRemove`                           |

`config.onRemove` is the delete counterpart. Removing a row (from the overflow `⋮` menu, per the CRUD "Delete & destructive" doc) now **always confirms first**, then — if `onRemove` is set — runs it with the row item. The row is dropped from the field value only when `onRemove` resolves successfully; returning `{ success: false }` or throwing keeps the row and surfaces an error, and the remove action is disabled while the request is in flight. Without `onRemove`, removal is value-only (the row is spliced out) but the confirmation is still shown.

Use `config.confirmRemove` to **name the resource** in the confirmation (what, how much, whether reversible). When omitted, a generic default confirmation is used.

```tsx
config: {
  onRemove: async (item) => {
    await api.delete(item.id)
    return { success: true }
  },
  confirmRemove: (item) => ({
    type: "critical",
    title: `Remove ${item.name}?`,
    msg: "They will lose access immediately. This cannot be undone.",
    confirm: { label: "Remove" },
  }),
}
```

### Custom row actions

`config.rowActions` adds trailing action buttons per row. It's resolved per row, so the actions can depend on the row's value — e.g. an Archive / Unarchive toggle driven by a hidden `archived` column. Each action's `onClick` receives `update` / `remove` helpers to mutate that row. The `remove` helper routes through the same confirm + `onRemove` path as the built-in remove, so custom "Delete" actions also confirm and persist:

```tsx
config: {
  columns: { archived: { hidden: true } },
  rowActions: (item) =>
    item.archived
      ? [{ icon: ArchiveOpen, label: "Unarchive", onClick: ({ update }) => update({ archived: false }) }]
      : [{ icon: Archive, label: "Archive", onClick: ({ update }) => update({ archived: true }) }],
}
```

### Other options

- `sortable` (default `true`) — drag rows by their handle to reorder.
- `canAddItems` (default `true`) — show the add button. Adding is blocked while an existing row is still invalid.
- `minItems` / `maxItems` — array-length validation; the add button hides at `maxItems`.
- `labels` — `addButton`, `create`/`update` (each `{ title, description }`) and `edit`/`remove` action labels. The create description doubles as the add button's tooltip; the update title as the icon-only edit action's tooltip.
- `onRemove` / `confirmRemove` — persist and confirm removals (see [Persisting removals](#persisting-removals)).
- `visualization` — `"editable-table"` (default) or `"list-view"` (see above).

### Auto-save

Any field accepts `autoSave: true` — a change to it saves the form (a debounced submit) without waiting for the Save button; other fields still save on submit. It's handy for an entities list so row edits persist as they happen. This is per-field and independent from the form-level `submitConfig: { type: "autosubmit" }` (which auto-saves on _any_ change). Auto-save submits the whole form, so it also persists pending changes elsewhere and validates every field.

```tsx
links: f0FormField.entitiesList({
  label: "Links",
  autoSave: true,
  schema: z.object({ title: z.string().min(1), url: z.string().url() }),
})
```

# Guidelines

## Design Best Practices

### When to use

- Any create or edit form that requires validation and a submit flow.
- Multi-step wizards — use `F0WizardForm` / per-section schemas with the same `f0FormField` API.
- Inline forms inside dialogs or sidepanels.
- Small, self-contained collections managed within the form — use `f0FormField.entitiesList` (e.g. links, options, line items). For large, filterable, paginated datasets use `OneDataCollection` instead.

### When NOT to use

- Simple one-field inputs with no submit action: use the standalone input component directly.
- Read-only displays: use a description list or data table instead.

### How to use

- Always embed validation rules near the field definition, not spread across other files.
- Use shortcut helpers for standard fields; reserve the base call for fields that need constraints not covered by shortcuts.
- Provide `helpText` on fields with non-obvious constraints.
- Use `row` grouping to lay out related fields side by side (e.g., first name + last name).

  }}
/>

## Accessibility

- All fields rendered by F0Form include ARIA labels derived from the `label` config.
- Error messages are linked to their field via `aria-describedby`.
- The form's submit button manages `aria-busy` during async submission.

## Opening a form imperatively — `forms.open`

`forms.open({ formDefinition, as })` opens a form from anywhere — no `useState` /
`useF0Form` / `formRef` wiring — and `await`s the outcome. Choose the
presentation with `as`: `"dialog"` for a single-screen form, or `"wizard"` for a
multi-step [`F0WizardForm`](?path=/docs/patterns-forms-f0wizardform--documentation).
Requires `<F0Provider>` to be mounted.

### As a dialog

Pass `mode: "dialog"` to open the form in a single-screen dialog:

```tsx
import { forms, f0FormField, useF0FormDefinition } from "@factorialco/f0-react"
import { z } from "zod"

const formDefinition = useF0FormDefinition({
  name: "add-member",
  schema: z.object({
    name: f0FormField.text({ label: "Name", minLength: 2 }),
    email: f0FormField.email({ label: "Email" }),
  }),
  defaultValues: { name: "", email: "" },
  onSubmit: async ({ data }) => {
    await save(data)
    return { success: true }
  },
})

const result = await forms.open({
  formDefinition,
  mode: "dialog",
  title: "Add team member",
  description: "They will receive an invitation email.",
})

if (result.submitted) {
  // result.data is the validated, typed form values
}
```

Behavior:

- **Submit succeeds** → the dialog closes and the promise resolves with
  `{ submitted: true, data }`.
- **Validation fails** → the dialog stays open with inline field errors.
- **Cancel / close button** → resolves with `{ submitted: false }`.

The dialog is **modal by default**, so an outside click or Escape won't dismiss
it (and discard in-progress input) — pass `modal: false` to allow that. The
dialog footer supplies the submit and cancel buttons (override their labels with
`labels: { submit, cancel }`), so you don't need `submitConfig.hideSubmitButton`
— `forms.open` sets it for you.

### As a wizard

Pass `mode: "wizard"` to open the same typed definition as a multi-step
[`F0WizardForm`](?path=/docs/patterns-forms-f0wizardform--documentation). Group
fields into steps with `sections` on the definition; `forms.open` resolves once
the **final** step submits:

```tsx
const result = await forms.open({
  formDefinition,
  mode: "wizard",
  title: "Add employee",
})

if (result.completed) {
  // result.data is the validated, typed full-form values
}
```

Behavior:

- **Final step submits successfully** → the wizard auto-closes and the promise
  resolves with `{ completed: true, data }`.
- **Dismissed / closed** → resolves with `{ completed: false }`.

If the definition uses `linkAfterLastStepSubmit`, the page navigates away and the
promise will not resolve.
