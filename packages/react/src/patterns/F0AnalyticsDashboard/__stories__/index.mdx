**Do's and don'ts**

<DoDonts
  do={{
    description: "Keep report filters global and predictable.",
    guidelines: [
      "Use one applied filter state for every participating item",
      "Keep filter labels consistent with the report's data vocabulary",
      "Persist only committed values",
    ],
  }}
  dont={{
    description: "Avoid mixing report and item scope.",
    guidelines: [
      "Don't present a dashboard filter when it changes only one item",
      "Don't persist temporary picker selections before Apply",
      "Don't silently retain values for removed filter definitions",
    ],
  }}

### Behavior

- WHEN <code>filtersValue</code> is provided → THEN the dashboard is controlled and waits for the parent to pass each committed value back.
- WHEN only <code>defaultFilters</code> is provided → THEN the dashboard owns subsequent committed filter state.
- WHEN Apply, Clear, chip removal, or a preset commits a change → THEN <code>onFiltersChange</code> receives the next applied state and participating items refetch with it.
- WHEN the picker closes without applying → THEN temporary selections are discarded.
- WHEN an item sets <code>useDashboardFilters=false</code> → THEN that item fetches with an empty filter object.
- WHEN <code>items</code> is empty → THEN the filter bar remains usable without a grid item fetch.

### Screen reader behavior

- The filter trigger names the applied filter labels, while a visible counter shows how many report filters are active.
- The filter popover has a dialog name, and filter options expose native checkbox semantics.
- Applied filter removal buttons include the filter label and selected value in their accessible names.
