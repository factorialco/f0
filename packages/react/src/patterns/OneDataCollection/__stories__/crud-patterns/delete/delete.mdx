# Delete pattern

Delete patterns cover destructive actions in a Data Collection. The confirmation
step must make the scope and consequence explicit before the user commits.

## When to use

- Use a Confirmation Dialog for single destructive actions.
- Use a scoped Confirmation Dialog for bulk destructive actions.
- Keep delete available as an explicit item or bulk action in every
  visualization. **Recommended** placement changes by view, but confirmation
  rules do not.
- When delete appears as a dialog header action, mark it as critical and keep it
  in the ellipsis menu. Any critical header action should force ellipsis, even
  when it is the only extra action.
- Name the affected resource or selection in the title whenever possible.
- Explain irreversible consequences in the body when data loss is involved.

## Guidance

Destructive actions should never depend only on iconography or placement. The
user should understand what will be removed and whether the change can be
reverted. This is especially important in bulk flows, where the risk of deleting
the wrong scope is higher.

Keep the confirmation concise. The goal is not to repeat the entire item
details, but to state enough context to prevent accidental deletion.

Table, List, Card, Kanban, and Editable Table all follow the same delete rule:
the user must confirm the affected item or selected scope before the destructive
change runs. In dialog headers, destructive actions should not become standalone
icon buttons. Mark delete as critical and keep it in the ellipsis menu so the
destructive action stays intentional and does not become a first-class visual
affordance.

In Card views, use the ellipsis because card click is a read affordance. Keep
delete in the ellipsis in both the card and the opened dialog, so the destructive
action stays consistent and separate from read. In Kanban views, use
selected-card Bulk Delete for destructive actions across multiple cards. Per-card
destructive actions may also be added via the card ellipsis, mirroring the Card
view rule so delete stays in the ellipsis. Moving cards updates state, never
destroys data. In Editable Table, use the row actions dropdown and keep
delete separate from editable cell interactions.
