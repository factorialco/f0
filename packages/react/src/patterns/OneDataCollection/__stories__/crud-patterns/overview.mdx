# Data collection / CRUD patterns

## Introduction

**CRUD** (Create, Read, Update, Delete) patterns describe the UX for
interactions with records from a Data Collection. They are product patterns that
consumers wire through collection actions, item actions, item click handlers,
bulk actions, dialogs, wizards, or routes.

Use these patterns to decide whether a collection interaction should stay in the
current surface, open a dialog, guide the user through a wizard, or navigate to a
dedicated page.

These stories are reference compositions, not visual mockups. Each scenario is
built with the same F0 layouts, headers, dialogs, wizards, and collection
components that production flows are expected to use.

## Scenario stories

The linked stories open as fullscreen scenarios and use Data Collection as the
interaction surface. They combine F0 layouts, headers, dialogs, wizards, and
collection components in the same way production flows should.

These stories assume the user has create, update, and delete permissions.
Production flows should hide or disable each action when the user lacks the
corresponding permission.

## CRUD by visualization

CRUD patterns are operation-first. The visualization changes the affordance, not
the rule. Use the **Recommended** pattern in each cell as the starting point, and
only move to an alternative when the resource or task needs it.

Kanban cards support drag, click-to-read, and selection at the same time. Drag
updates lane and state, click opens the read dialog, and selection drives bulk
actions. These three interactions must not interfere with each other: the card
click target stays whole, the drag handle is the card itself, and the selection
checkbox stops propagation so toggling it never opens the card.

## Pattern boundaries

Use a default dialog for standard create, read, and update flows. Update should
mirror create whenever possible: reuse the same form structure, field order,
validation rules, and terminology when users update the same information.

Use a right-position dialog as an alternative update pattern when keeping the
collection visible materially helps the task. This is most useful when users
need to compare against nearby items or stay anchored to the current collection
context while editing.

Read can stay entirely in a dialog when the resource is lightweight enough to
fit a focused overlay. A right dialog can also work as a preview step when the
resource has a meaningful full-page destination, but it does not have to route
to a page in every case.

The dialog-to-page behavior is portable. Any compatible default dialog or
right-position read dialog can use **Open Details** as its primary
action when the dialog is intentionally a reduced preview and the resource has a
durable full-page destination. This logic applies across Table, List, Card, and
Kanban when the target page provides deeper structure.

Move to a table to page or list to page pattern when the resource needs
persistent navigation, deeper structure, page-level actions, or a durable
destination beyond a quick inspection flow.

Use bulk update when the visualization provides clear multi-selection
affordances and users need to apply one change to a selected set of items.
Bulk actions are not Table-only. Table checkboxes are the clearest example, but
Card, Kanban, List, or Editable Table can use bulk update when selected items
remain visible, countable, and reviewable before confirmation. If the scope is
ambiguous, use item-level update or a default dialog instead.
