# Data collection / CRUD patterns

## Introduction

**CRUD** (Create, Read, Update, Delete) patterns describe the UX for
interactions with records from a Data Collection. Use them to decide whether an
interaction should stay in the current surface, open a dialog, guide the user
through a wizard, or navigate to a dedicated page.

The **Pattern decision table** below is the canonical reference. Later sections
expand on individual operations, views, and edge cases.

**A note on width.** Dialog containers (Default dialog, Right dialog, Wizard
dialog) are all backed by the same `F0Dialog` primitive with a configurable
`width` prop accepting `sm`, `md`, `lg`, or `xl`. Pick width based on content
density, not pattern type. Most flows use `md`. Use `sm` for the **Dialog to
page** preview — it is intentionally compact since the full page is the
durable destination — and `lg` or `xl` for content-dense forms or wizards.

## Container anatomy

Each named pattern is a specific configuration of `F0Dialog`. The differences
are in `position`, `width`, and the actions inside — not in the underlying
component.

## Pattern boundaries

Use a default dialog as the baseline for create, read, and update. Update should
mirror create — same form structure, field order, validation, and terminology. A
right dialog is an alternative for Table and List update when keeping the
collection visible materially helps (comparing nearby items, staying anchored to
context).

Read can stay in a dialog when the resource is lightweight. In Table and List,
a right dialog is also valid when keeping the collection visible helps the user
scan or compare adjacent items. The **Dialog to page** flow uses a right dialog
as a preview that opens a full page when the resource has a durable
destination, tabs, deeper structure, or page-level actions — any compatible
default or right dialog can use **Open Details** as its primary action. Bulk
update is valid across visualizations whenever multi-selection scope is
visible, countable, and reviewable; if scope is ambiguous, use item-level
update instead.

Kanban cards support drag, click-to-read, and selection simultaneously. Drag
updates lane and state, click opens the read dialog, and the selection checkbox
stops propagation so toggling it never opens the card.

## View-specific exceptions

Each view follows the Pattern decision table; these are the cases where it
doesn't.

- **Table:** right dialog for read or update when row comparison or scanning
  helps the task.
- **List:** right dialog for read or update only when list context remains
  useful while reading or editing.
- **Card:** click opens read first; edit is an explicit action inside the read
  dialog.
- **Kanban:** lane create for state-scoped resources; drag for state updates;
  bulk action bar for cross-lane scope.
- **Editable Table:** inline cell edit for field-level updates; add-row for
  lightweight new rows; row dropdown for delete.
