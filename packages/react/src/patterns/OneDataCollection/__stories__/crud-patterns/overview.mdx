# Data collection / CRUD patterns

## Introduction

**CRUD** (Create, Read, Update, Delete) patterns describe the UX for
interactions with records from a Data Collection. Use them to decide whether an
interaction should stay in the current surface, open a dialog, guide the user
through a wizard, or navigate to a dedicated page.

## CRUD by view

Open a fullscreen scenario that shows create, read, update, and delete
coexisting inside one view. Use these to validate how the four operations
behave together in real product surfaces. Editable Table is a specialized
Table mode for field-level updates; most Table rules still apply.

## Pattern decision table

The Pattern decision table below is the canonical reference. Later sections
expand on individual operations, views, and edge cases.

common Read interaction observed across the product. Dialog-based reads are
reserved for cases where the resource is small enough to be fully understood
in a dialog and no durable page destination exists.

**Default Dialog**, **Right Dialog**, **Wizard Dialog**, and **Confirmation
Dialog** are dialog containers. **Dialog to Page**, **Table to Page / List to
Page**, and **Bulk Update** are flows built on top of those containers.

**A note on width.** Dialog containers are all backed by the same `F0Dialog`
primitive with a configurable `width` prop accepting `sm`, `md`, `lg`, or
`xl`. Default Dialog uses `md`. Wizard Dialog typically uses `md`; reach for
`lg` only when step content truly requires it. Confirmation Dialog uses `sm`.
Dialog to Page typically uses `sm` since the full page is the durable
destination. Stay between `sm` and `md` for CRUD flows — avoid `lg` unless
content density requires it, and avoid `xl`.

## Container anatomy

Each named pattern is a specific configuration of `F0Dialog`. The differences
are in `position`, `width`, and the actions inside — not in the underlying
component. Container anatomy covers dialog containers only. Non-dialog
patterns (Table to Page / List to Page, Editable Table inline edits, Kanban
drag) live on each view's own documentation page.

## Pattern boundaries

Use a Default Dialog as the baseline for create, read, and update. Update
should mirror create — same form structure, field order, validation, and
terminology.

In Table and List, a Right Dialog is a valid alternative for create, read, or
update when keeping the collection visible materially helps (comparing nearby
items, scanning adjacent rows, staying anchored to context). This is the only
place outside the Pattern decision table that states the rule — keep it here
when reasoning about new flows.

The **Dialog to Page** flow uses a Right Dialog as a preview that opens a full
page when the resource has a durable destination, tabs, deeper structure, or
page-level actions. Any compatible Default or Right Dialog can use **Open
Details** as its primary action.

Bulk Update is valid across visualizations whenever multi-selection scope is
visible, countable, and reviewable; if scope is ambiguous, use item-level
update instead.

## View behaviors

How each view applies the Pattern decision table by default.

- **Table:** route to a page by default for read. Right Dialog only when row
  comparison materially helps the task. Default Dialog only when the resource
  is small and no page destination exists.
- **List:** route to a page by default for read. Right Dialog only when list
  context remains useful while reading. Default Dialog only when the resource
  is small and no page destination exists.
- **Card:** click opens read first; edit is an explicit action inside the read
  dialog or from a card item action (ellipsis menu).
- **Kanban:** lane create for state-scoped resources; drag for state updates;
  bulk action bar for cross-lane scope. Kanban cards support drag,
  click-to-read, and selection simultaneously — drag updates lane and state,
  click opens the read dialog, and the selection checkbox stops propagation
  so toggling it never opens the card.
- **Editable Table:** inline cell edit for field-level updates; add-row for
  lightweight new rows; row dropdown for delete.
