# Data collection/Visualizations

## Introduction

Visualizations are the way to display the data in the collection.

Data collection abstracts the data (including, filtering, sorting, grouping,
etc.) and the visualization, of the data at item level (visualization) and
[cell level](?path=/docs/data-collection-cell-types--docs).

## Per-visualization source override

Each entry in the `visualizations` array accepts either a filter-branch
(`filters` / `presets`, derived from the top-level `source`) **or** a
`source` override that replaces the top-level source while that
visualization is active. The two branches are mutually exclusive at the
type level — `source` cannot be combined with `filters` / `presets`, and
vice versa.

```tsx
<OneDataCollection
  source={topLevelSource}
  visualizations={[
    // Filter branch — derives rows from topLevelSource
    {
      type: "table",
      options: tableOptions,
      filters: {
        /* ... */
      },
    },

    // Source-override branch — replaces topLevelSource for this viz
    {
      type: "graph",
      source: graphSource,
      options: { nodeAdapter, renderNode },
    },
  ]}
/>
```

### When to reach for it

Use a per-viz `source` override when the shape behind one visualization
differs materially from the rest of the collection — for example a flat
paginated list for the table and a hierarchical / DAG shape for the graph.
The override source can carry its own `filters`, `sortings`,
`navigationFilters`, `selectable`, `bulkActions`, and `itemActions`
independently of the top-level source.

### Chrome routing

While an override-source visualization is active, the collection chrome
routes to the **override source**:

| Chrome surface  | Routes to override source |
| --------------- | ------------------------- |
| Filter bar      | yes                       |
| Search          | yes                       |
| Presets         | yes                       |
| Bulk-action bar | yes                       |
| CSV export      | yes                       |
| Item actions    | yes                       |

The top-level source remains mounted and continues to drive any
non-override visualizations.

### Pagination guard

The graph visualization rejects paginated override sources at mount in
eager mode (no `loadChildren`) and emits the same
`EAGER_REQUIRES_NO_PAGINATION_MESSAGE` it raises for paginated top-level
sources. Lazy mode (`loadChildren` defined) tolerates pagination on the
override source the same way it does on the top-level source.

### Dev warning

When an override source is detected, OneDataCollection emits a
`console.warn` (dev-only) the first time the override viz becomes active.
This keeps the surface discoverable while it remains alpha and helps
catch accidental misuse of `source` as a substitute for `filters`.

See the
[Per-viz Source Override](?path=/story/data-collection-visualizations-per-viz-source-override--with-override-source-eager)
stories for runnable examples and the `__tests__/` directory for the
pagination guard and type-level mutex coverage.
